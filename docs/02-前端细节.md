# AI 生成器平台 — 前端细节

> 本文档定义前端（Vue 3 + Vite + TS 纯静态 SPA）的实现细节：**实时流、WebLLM 首过、预览隔离、模型选择器、用户评价、管理后台、本地开发配置**。
> 跨服务约定见《项目总览》；后端接口与鉴权见《业务端细节》；生成链路见《AI 核心端细节》。
> 全文以「当前方案」表述，不含历史修订记录。

---

## 1. 技术栈与总体形态

- **框架**：Vue 3 + Vite + TypeScript（纯静态 SPA，非 SSR）。
- **状态管理**：Pinia（对话会话、当前模型、用户态、Trace 缓冲）。
- **UI 库**：Tailwind CSS + Naive UI / Element Plus；**浅色简洁**视觉基调。
- **路由**：Vue Router；用户前台 + `/admin` 管理后台为同一应用内路由，彼此隔离、共享登录态。
- **实时通道**：SSE（`EventSource`），不使用 WebSocket（见《项目总览》§5）。
- **本地运行**：`vite.config.ts` 设 `server.port = 7100`（`host: true` 允许域名/局域网访问），`server.proxy` 将 `/api` 与 `/auth` 代理到 `http://localhost:7101`（业务服务；hosts 中 `seedapi.huzhen.net.cn` 亦指向 127.0.0.1，可直接 `seedapi.huzhen.net.cn:7101` 访问后端做联调/测试）。前端经 `http://seedai.huzhen.net.cn:7100` 访问，同源代理使 Cookie 自动携带（见 §11）。
  - 业务 / AI 服务若**本地直跑**（不走 docker），务必用 `python backend/business/app/main.py` / `python backend/ai_service/app/main.py` 启动——两处 `main.py` 的 `__main__` 块已把端口锁定为 **7101 / 7102**（读取 `business_api_port` / `ai_service_port`）。**切勿用裸 `uvicorn app.main:app`**，否则回退到 uvicorn 默认 **8000**，与前端代理端口对不上。docker 路径由各自 Dockerfile 的 `uvicorn ... --port 7101/7102` 启动，不受影响。

## 3. 实时传输：SSE

前端与业务服务之间的实时生成流采用 **SSE（Server-Sent Events）**——单向服务器推送，天然适配「思考过程 + 产物」事件流。

- **连接生命周期**：用户每次发起提问时，新建一个 `EventSource` 连到业务服务 `GET /api/chat?projectId=...`（带 HttpOnly Cookie 自动鉴权）；本次提问（含多 agent 协作与最终产物）推送完毕后，业务服务发 `done` 事件并**结束响应**（连接关闭）；用户中断时 `eventSource.close()` 主动关闭。
- **鉴权**：复用 HttpOnly Cookie 中的 JWT（同源自动带），业务服务在处理前校验。
- **登录门禁（已落地）**：进入即展示主页（对话页），右上角显示「登录/注册」按钮（已登录则显示**昵称** + 设置 + 退出）。**提交对话时**若未登录，弹出登录/注册模态框（`AuthPanel.vue`）拦截，登录/注册成功后由 `watch(user)` **自动关闭**并继续；密码框带可见性切换。注册表单含可选「昵称」字段（留空默认同用户名）。右上角「设置」打开 `SettingsPanel.vue` 模态面板，可修改昵称 / 邮箱 / 密码（改密码需验旧密码）。登录态见 `src/api/auth.ts`，由 `stores/auth.ts`（Pinia）+ `composables/useAuth.ts` 管理，`AuthUser` 含 `nickname` 字段。
- **鉴权失败主动弹窗（已落地，v0.2.1）**：任何请求遇到 401 / `Missing authentication` / `Invalid or expired token` 时，前端**主动弹出登录框**，而不是只显示一行报错。
  - 全局登录弹窗开关在 `stores/auth.ts`：`loginOpen` + `openLogin()/closeLogin()`；`login()/register()` 成功后自动 `loginOpen=false`。`AuthPanel.vue` 改为由该 store 驱动（自身 `v-if="store.loginOpen"` 显隐，不再由父组件 `v-if` 控制），`TopNav.vue` 始终挂载一份（全局唯一）。
  - 数据接口（`api/projects.ts` 的 `j()`）：`fetch` 返回 `401` 时调用 `notifyAuthRequired()` 并抛出「登录已失效，请重新登录」。
  - SSE 对话（`api/chat.ts` 的 `startChat` error 监听）：后端鉴权失败会下发 `error` 事件（`code=AUTH_REQUIRED`、`message="Missing authentication"`），前端匹配该 `code` 或正则（missing authentication / not authenticated / 未登录 / invalid or expired token / 401）后调用 `notifyAuthRequired()` 并提示「登录已失效，请重新登录」。`notifyAuthRequired()` 为 `stores/auth.ts` 导出的模块级函数，供非组件模块（api/*）直接调用。
  - 注：`EventSource` 读不到 HTTP 状态码，故后端 `/api/chat` 鉴权失败**不再返回 JSON 401**，而是下发上述 SSE `error` 事件（见《业务端细节》§5）。
  - **上游 / 服务器错误明确提示（已落地，v0.2.2）**：SSE `error` 监听按后端下发的 `code` 分流——`UPSTREAM_ERROR`→「AI 服务暂时不可用，请稍后重试」、`RATE_LIMITED`→「请求过于频繁，请稍后再试」、`AUTH_REQUIRED`→弹登录框，其他有 `message` 则显示之；**仅当 `error` 事件无 data（真正网络中断 / 连接被重置）才提示「连接中断，请检查网络或重新登录」**。这样把“服务端 5xx 崩了”与“网络真断了”明确区分，避免误导用户去查网络。
- **事件协议**（`event:` 区分类型，`data:` 为 JSON）：
  - `token` —— Coder 产出的 HTML 字符流（最终产物，进预览）
  - `think` —— 各 agent 的思考 / 推理文本（Planner 规格、Router 决策、Reviewer 结论）
  - `node`  —— agent 节点进入 / 离开（前端高亮当前阶段，按步精确实时反馈）
  - `plan`  —— 大计划 / 目标特殊节点（Planner 产出的结构化规格 title/goal/steps），前端渲染为「计划 / 流程」卡片，区别于普通 think 文本
  - `error` —— 错误（随后结束）
  - `done`  —— 结束（随后断连）
  - `aborted` —— 主动取消确认（C1 级联取消链路，见 §9）
  - `degraded` —— 模型降级标记（2-C，前端显示「已降级」小标）

> 思考事件 `think`/`node` 与产物流 `token` 分离：前端可同时显示「思路」与「成品」。

## 4. 对话页三区（气泡 / 思考面板 / 预览）

对话页由三个联动区域组成：

1. **对话气泡区（MessageBubble）**：用户消息与 AI 文本回复，按 `Message`（user / assistant）渲染。
2. **思考过程面板（ThoughtTrail）**：实时展示 `think`/`node` 事件 + `plan` 计划卡片，呈现 AI 规划 / 推理 / 评审思路（分步时间线，每 agent 节点一步）；简单 Skill（写代码 / 文档）仅发 `node:dispatch` + `token`，无多余 `think` 噪音。
3. **产物预览区（PreviewPane）**：`token` 流累积为 HTML，经 §7 的隔离方式渲染；产物最终版本上传 COS 后 `iframe` 指向线上直链。

三区随 SSE 事件滚动同步更新；会话结束后由 `RatingWidget` 收集评价（§8）。

## 6. WebLLM 浏览器内首过（客户端优先）

> **⚠ 未实装**：前端 `useWebLLM` 未接入，模型选择器无 WebLLM 选项；当前所有生成走云端 SSE。本节为规划，落地见后续决策。

利用用户本机算力（WebGPU）在浏览器内跑一个小模型，对**简单需求做「首过」生成 / 规划**，从而**降低云端 token 消耗、缩短首字延迟、增强隐私**。云端 Orchestrator 仍是主力，WebLLM 是其本地加速 / 隐私 / 降成本补充通道。

- **选型**：浏览器端 LLM 运行时用 **@mlc-ai/web-llm**（WebAssembly + WebGPU，支持 Llama-3-8B-Instruct / Qwen2.5-7B-Instruct 等 GGUF 量化模型）。模型首次使用需从 CDN 下载权重（约 2–5GB），后续走浏览器 Cache Storage 缓存；仅 Chrome/Edge 等支持 WebGPU 的浏览器可用。
- **定位（MVP 采用 A + 单模型 + 预取）**：
  - 选中「本地 WebLLM」后，前端用 WebLLM **本地**完成 Planner 规划 +（简单站点）Coder 生成，全程本地、用与云端**相同的 `think`/`token`/`node` 事件帧**在本地产出，产物直接进 iframe 预览；
  - 仅当用户选择云端模型、或本地生成失败 / 超时 / 不支持 WebGPU，才回退到业务服务 → AI 服务 SSE。
- **事件一致性**：WebLLM 本地产出的 `think`/`node`/`token` 与云端 SSE 帧**完全同构**，前端「思考过程面板 + 预览」无需区分来源。
- **预取与加载**：`useWebLLM` 封装 `warmup()`（首屏空闲 / 用户 hover 模型选择器时预取权重到 Cache Storage）+ `loadModel()`（命中缓存秒开）；锁定单一固定小模型（推荐 `Qwen2.5-7B-Instruct-q4f16_1-MLC` 档）；不支持 WebGPU 时选项灰显并提示回退云端。
- **限制**：WebLLM 仅能跑小模型，复杂 / 大页面质量不如云端；首下载慢、占本机内存；因此**云端 Orchestrator 不被替代**，WebLLM 仅作首过通道，完全在前端、不触及业务 / AI 服务的模型 Key。

## 7. 预览与隔离（iframe + sandbox + CSP + COS 直链）

- **主路径（A2）**：生成的单文件 HTML **上传腾讯云 COS**，预览走 **COS 默认访问域名直链**（`https://seedhtml-1252059540.cos.ap-guangzhou.myqcloud.com/{base_path}/{user_id}/{site_id}/{version}/index.html`）。`iframe` 指向该 URL——与主站 `seedai.huzhen.net.cn` **天然跨域隔离**（COS 域不同源，同源攻击面隔离）。详见《AI 核心端细节》§11。
- **降级兜底**：COS 投递失败或本地离线时，降级回 `<iframe srcdoc>` 直接注入。
- **安全属性**：
  - `iframe` 加 `sandbox="allow-scripts"`（不含 `allow-same-origin`），防生成脚本访问主站 Cookie / 存储。
  - **CSP（当前）**：COS 当前未开 CDN，**不在预览对象打 `Content-Security-Policy` 响应头**，仅靠 `sandbox` 隔离（决策 #32）；待绑定 `seedhtml.huzhen.net.cn` + CDN 后由回源重写注入严格 CSP（`default-src 'none'` + 白名单 CDN / 字体）。
  - srcdoc 兜底路径下，同样加 `sandbox="allow-scripts"` + 内联 CSP `<meta>`（离线无外链风险）。
- **增强**：含 React / 多文件 / 需构建的页面，增强阶段上 Sandpack（浏览器内打包）；需真实 Node 运行暂不做。

## 9. 取消与重试（AbortController + 级联取消）

- **前端取消**：用户点「停止」→ `StopButton` 调 `eventSource.close()` 断开 SSE。
- **级联取消（C1）**：业务服务感知连接关闭 → 经 Redis 广播 `cancel:<traceId>` → AI Worker 监听到标记中断当前生成（asyncio task cancel / callback 抛 `GenerationAborted`），释放槽位并停止计费；`done` 前可多发一帧 `aborted: true`。
- **重试 / 重连（离线继续）**：AI Worker 独立常驻后台，进度写入可回放 Redis Stream（业务端 §5）。前端用 `sessionStorage` 记录 active `trace_id`；刷新页面 / 重连时复用同一 `trace_id` 重开 SSE，业务命中"续接"后全量回放 + 续接实时进度（带 `after` 可增量续传），从而恢复完整对话流程——用户离线后后台继续生成，重连即可续上。落库按 `trace_id` 幂等（同一 trace 不重复插消息）。

## 10. 管理后台（/admin 路由）

- 作为 Vue 内 `/admin` 路由，与用户前台共享登录态、彼此隔离。
- **权限（RBAC 三级，见《业务端细节》§3）**：`super_admin`（全部：后台查看 + 控制面执行 + 用户 / 角色管理）/ `admin`（仅查看后台）/ `user`（不可进）。
- **看板内容**：
  - 运行指标（服务状态 / QPS / 延迟 / 错误率 / 模型用量），经 SSE 实时推送（轮询兜底）。
  - **AI 质量**标签页：6+1 类指标（效率 / 准确性 / 消耗量 / 回滚率 / 用户回馈 / 健康度 + RAG 检索质量），见《业务端细节》§8。**（⚠ 未实装：当前仅基础运行指标 uptime/请求/模型用量；6+1 维度依赖 Trace/UsageLog/Feedback 表，均未建）**
  - **回放**：按 `trace_id` 拉 `TraceEvent` 序列还原「用户说 / AI 想 / AI 做 / 操作」全过程（复用对话页组件渲染）；`user` 仅能回放自己会话。**（⚠ 未实装：`TraceEvent` 表未建，`/admin/traces/:traceId` 为 TODO）**
  - **控制面**：`super_admin` 可见可执行的手动启停 / 扩缩容（Orchestrator，docker-compose 封装，二次确认）；`admin` 仅查看。**（⚠ 占位：当前 `/admin/scale`、`/admin/stop` 仅返回 ack stub，未接 DockerComposeOrchestrator）**
- **数据来源**：指标与回放均由业务服务聚合（MySQL + Redis），前端仅消费 `/admin` 相关端点。

## 11. 本地开发配置（vite.config.ts）

```ts
// vite.config.ts（节选）—— 注意:这是前端唯一的 Vite 配置来源
export default defineConfig({
  server: {
    host: true,                      // 允许通过域名(如 seedai.huzhen.net.cn)/局域网 IP 访问,否则只绑 localhost
    port: 7100,                      // 对齐 hosts / CORS
    allowedHosts: true,              // Vite 5.4+ 默认拦截非 localhost 的 Host 头,本地 dev 关掉该检查
    proxy: {
      '/api': {
        target: 'http://seedapi.huzhen.net.cn:7101',  // 业务服务(127.0.0.1:7101)
        changeOrigin: true,
      },
      // 登录/注册等鉴权接口也同源代理,Cookie 随请求自动携带
      '/auth': {
        target: 'http://seedapi.huzhen.net.cn:7101',
        changeOrigin: true,
      },
      // 管理后台接口(指标 SSE / 用户管理 / 控制面)同源代理(v0.5.0 新增)
      '/admin': {
        target: 'http://seedapi.huzhen.net.cn:7101',
        changeOrigin: true,
      },
    },
  },
})
```

> ⚠️ **配置唯一来源是 `vite.config.ts`**：切勿生成 `vite.config.js` 副本——Vite 按查找顺序优先加载 `.js`，会**静默覆盖** `.ts`，导致 `host` / `allowedHosts` / `/auth` 代理等配置全部不生效（曾因此出现域名被 403 拦截、登录接口 404）。`.js`/`.d.ts` 已在 `.gitignore` 忽略。

- `host: true` + `allowedHosts: true`：使 `http://seedai.huzhen.net.cn:7100` 可访问（否则 Vite 仅绑 localhost，域名访问要么连不上、要么被 host 检查 403）。
- `server.port = 7100` 与 hosts 中 `seedai.huzhen.net.cn`、以及 `CORS_ORIGINS` 的 `http://seedai.huzhen.net.cn:7100` 完全一致。
- `/api` 与 `/auth` 均代理到 `seedapi.huzhen.net.cn:7101`（业务服务；hosts 里 `seedapi.huzhen.net.cn` → 127.0.0.1）。前端用**相对路径** `/api`、`/auth`，浏览器看到的源始终是 `seedai.huzhen.net.cn:7100`(同源),Cookie 由后端下发后落在 `seedai.huzhen.net.cn` 上,后续请求自动携带,经代理透传到业务服务——无需跨域、无 CORS 顾虑。
- 预览域 `iframe` 直接指向线上 COS 默认域,不在此代理。

## 12. 本地 bare 域名访问(无端口,即 80)
若希望用 `http://seedai.huzhen.net.cn/`(无端口)与 `http://seedapi.huzhen.net.cn/` 直接访问(而非 `:7100`/`:7101`),需在本地 80 端口加一层反代(本机 nginx):
- 配置:`deploy/nginx-local-dev.conf`(`seedai` 域 → `/` 反代前端 7100、`/api`+`/auth` 反代业务 7101;`seedapi` 域 → 全部反代业务 7101;均含 SSE 缓冲关闭与 vite HMR websocket 升级头)。
- 前置:本机已安装 nginx(如 `winget install -e --id nginx` 或 `choco install nginx`),hosts 已映射两域名到 127.0.0.1。
- 启动:**以管理员**运行 `deploy/start-local-proxy.bat`(内部 `nginx -c deploy/nginx-local-dev.conf -p deploy`);停止 `nginx -c deploy/nginx-local-dev.conf -s stop`(同样管理员)。
- 注意:前端需先 `npm run dev`(监听 7100),bare 域名 `seedai.huzhen.net.cn/` 才会返回 200;否则反代返回 502。后端 7101 已起时 `seedapi.huzhen.net.cn/auth/me` 即可通。
- 若前端改走 bare 域名(`http://seedai.huzhen.net.cn`,无端口)或 HTTPS,`CORS_ORIGINS` 已含这两种 origin,Cookie 域需用 `cookie_domain=seedai.huzhen.net.cn`(生产 `.env.production` 已设);本地 http 开发保持 `cookie_domain` 为空(按宿主 host-only)即可。
