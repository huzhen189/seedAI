# v0.8.3 · 安全模块「短语/上下文感知」匹配（防误拦）

> 状态：✅ 已实施（2026-07-21） · 未 push · 待用户自启 7101/7102 端到端验证

## 0. 背景与遗留风险

v0.8.2 把安全关键词抽到 `common.py` 单一来源，但 `run_safety` 仍是**纯子串匹配**（`kw in text`）：

- 「帮我做一个**支付**页面」「生成一个**密码**输入框」「加一个**导出**按钮」「帮我写一段**删除**多余注释的代码」这类**正常建站需求**会被 `支付/密码/导出/删除` 误判为 critical / high，**直接拦截或强塞二次确认**，严重影响体验。
- 同时存在一个连带 bug：`删除用户` 因包含 `删除` 被错误升级为 critical（应为 high 确认）。

本次目标：**把软性关键词升级为短语/上下文感知**，保留真正破坏性短语的硬拦截。

## 1. 设计

### 1.1 关键词二分（common.py）

| 类别 | 含义 | 行为 |
|------|------|------|
| `SAFETY_HARD_KEYWORDS` | 真正破坏性/滥用**短语**（清空数据库、rm -rf、drop table、sql注入、删除数据库、导出数据库、删除所有…） | 命中即 critical 硬拦截，**不受语境影响** |
| `SAFETY_SOFT_CRITICAL/HIGH/MEDIUM` | 会出现在正常需求里的软词（支付/密码/导出/删除/发布/改密码/权限…） | 命中后查语境，语境命中则**中和（不拦截）**，否则按原严重度处理 |

> HARD 新增 `导出数据库`/`export database`/`备份数据库`/`删除所有`/`删除全部`，确保「导出数据库」这类危险短语即便带「帮我」也始终拦截。

### 1.2 中和判定（safety.py `_is_neutralized`）

SOFT 关键词命中后，满足以下任一即中和：

1. **建设性前导**：关键词前 `_LEAD_WINDOW=12` 字内有前导词。
   - critical / medium 用完整集 `CONSTRUCTIVE_LEADS`（含 `帮我`/`请帮我`/`需要`/`想要`…）；
   - **high 用严格集 `STRICT_LEADS`**（仅 `做一个`/`开发一个`/`加一个`/`写`/`做`/`开发`… 做功能/页面类），**避免「帮我删除用户」「帮我改密码」被误中和**。
2. **UI / 代码语境**：关键词前后 `_CTX_WINDOW=10` 字内有语境词（`页面`/`按钮`/`输入框`/`表单`/`组件`/`注释`/`函数`/`代码`…）。

中和的关键词记入 `SafetyResult.neutralized_tags`（仅观察，不拦截）。

### 1.3 重叠降级

`删除` 等 critical 裸词若是某 HIGH/MEDIUM 短语（如 `删除用户`）的子串，归并到该短语严重度（→ high），**不再误升 critical**。

## 2. 改动文件

- `backend/ai_service/app/intent/common.py`
  - `SAFETY_CRITICAL/HIGH/MEDIUM_KEYWORDS` → `SAFETY_HARD_KEYWORDS` + `SAFETY_SOFT_CRITICAL/HIGH/MEDIUM` + `CONSTRUCTIVE_LEADS` + `STRICT_LEADS` + `SAFETY_UI_CONTEXT`。
  - HARD 增补 DB 导出/清空类短语。
- `backend/ai_service/app/intent/safety.py`
  - 导入更新；`SafetyResult` 新增 `neutralized_tags` 字段（向后兼容，默认空）。
  - 新增 `_is_neutralized(text, kw, severity)`。
  - `run_safety` 重写为：HARD 硬拦截 → SOFT 分严重度收集 + 语境中和 + 重叠降级 → 取最高严重度。
- 无前端改动（`SafetyResult` 字段为增量，proxy/pipeline 无需改）。

## 3. 验证（28/28 用例通过）

`py_compile` 全部通过；`grep` 确认无旧常量残留引用。逻辑自测覆盖：

- **中和 → low（原误拦）**：支付页面、密码输入框、导出按钮、删除注释、转账页面、支付表单、修改卡片布局、发布功能页面、删除用户功能模块 …
- **HARD → critical（仍拦截）**：清空数据库、rm -rf、drop table、sql注入、删除数据库、**导出数据库（含「帮我导出数据库」）**、删除所有 …
- **high 管理动作（不放过）**：帮我删除用户、帮我改密码、帮我封禁、帮我修改权限、发布到生产、导出服务器日志、帮我发布网站 …
- **重叠降级**：删除用户并移除权限 → high（非 critical）。
- **项目约束**：`project_constraints=['deploy']` 命中仍 critical（Tier 2 不受影响）。

## 4. 影响与注意事项

- **不拦截范围扩大**：正常「做X功能/页面」类需求不再被安全层误拦，体验提升。
- **仍保守的边界**：`帮我删除用户`/`帮我改密码`/`帮我发布` 等真实管理危险动作仍走 high 二次确认（因 high 只用严格前导中和）。
- **HARD 永远拦截**：破坏性短语不受任何语境中和。
- **统计**：`neutralized_tags` 可供后续在管理后台「系统分析」展示「被语境中和的软词」，评估误拦改善效果（可选增强，本次未接）。

## 5. 后续可选增强（未做）

- 安全匹配可进一步升级为「短语/否定词感知」（如排除「不要/禁止/如何防范」等前导），但当前建设性前导 + UI 语境已覆盖绝大多数误拦场景，性价比不高，暂缓。
