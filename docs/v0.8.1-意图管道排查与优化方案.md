# v0.8.1 意图管道排查与优化方案

> 对照 `v0.8.0-intent-pipeline-refactor.md` 逐文件审查 ai_service 前端/后端实现，列出隐患与优化点。
> **状态：用户指令「帮我都改了罢」→ 全部 P0/P1/P2 已落地（见文末「落地状态」）。**

---

## 🚨 一、最严重：v2 管道决策结果未接入执行链路（汇总器形同虚设）

### 现象
`core/queue.py` 第 371 行：
```python
skill_name = skill or skill_for(intent["level1"], intent["level2"]) or "explain"
```
Worker 用 `skill_for(level1, level2)` **重新查表**计算 skill，完全丢弃了 `detect_intent_v2` 返回字典里的：
- `decision`（`route`/`block`/`confirm`/`options`/`fallback`）
- `selected_skill`（汇总器算出的置信度最优技能）
- `requires_confirm` / `risk_level`（安全模块结果）
- `plan`（含 options 候选技能列表）

### 后果（逐条对应文档设计的失效）
1. **安全拦截失效（critical）**：`safety.run_safety` 命中「删除/支付/清空」返回 `decision="block"`，但 Worker 不读 decision，仍用 `skill_for("learn","casual")`→`explain` 照常执行。文档承诺的"高风险直接拦截"是**死代码**。
2. **二次确认失效（high）**："发布/上线/deploy" 返回 `decision="confirm"`，Worker 不处理，直接执行，没有确认 UI。
3. **多选项失效（options）**：低置信度时 `decision="options"` 带候选技能列表，Worker 不读，退回单 skill（`skill_for`）。
4. **置信度路由被绕过**：`run_tools()` 算出的置信度排序、builder→requirement 状态路由全部被忽略；Worker 自己又**重写一份** `if skill_name=="builder_agent" and proj_status in draft/planning` 路由（重复逻辑、单一来源被破坏）。

### 修复方向（P0）
- Worker 改为消费管道输出：`skill_name = skill or intent.get("selected_skill") or "explain"`，并 `switch(intent["decision"])`：
  - `block` → 发 `block` SSE 事件后 done，不执行。
  - `confirm` → 发 `confirm` SSE 事件，等前端确认回传（带 `confirmed=true` 重发）后再执行。
  - `options` → 发 `options` SSE 事件（候选来自 `plan[].skills` 或 `intent["plan"]`），用户选完带 `skill=` 重发。
  - `fallback`/`route` → 正常走 `selected_skill`。
- `router.detect_intent_v2` 已正确返回这些字段，只需 Worker 接上。

## 二、前端无消费端（决策链路断在后端之前）

### 现象
- `frontend/src/types.ts` 的 `IntentEvent` 缺：`decision` / `risk_level` / `requires_confirm` / `plan` / `selected_skill`。
- `core/runner.py` 的 `ev("intent", ...)` 只透传 `level1/level2/labels/confidence/industry`，**丢失 decision 与 risk**。
- `chat.ts` 监听了 `options` 事件，但**只接 `requirement_agent` 自己发的 options**，不接管道级 `decision="options"` 触发的 options。
- 没有 `block` / `confirm` 事件的监听与 UI。

### 后果
即使 Worker 接上 decision，前端也收不到、不会弹确认/拦截框。

### 修复方向（P0/P1）
- `IntentEvent` 增加 `decision`/`risk_level`/`requires_confirm`/`plan`/`selected_skill` 字段。
- `runner.ev("intent", ...)` 透传上述字段。
- 前端新增：`onBlock`（高危拦截提示，不可继续）、`onConfirm`（二次确认弹框，确认后带 `confirmed=true` 重发）、统一 `options`（管道级 + requirement_agent 级合并为同一弹框）。

## 三、上下文模块 Chroma 分支是死代码（Bug）

### 现象
`core/queue.py` 第 326-332 行给消息建向量索引：
```python
idx = msg.get("_msg_id") or (conversation_id * 1000 + i)
index_message(idx, conversation_id, ...)
```
索引用的 `idx` 是**临时变量，没有回写进 `messages` 对象**。

`intent/context.py` 第 47 行：
```python
relevant = [m for m in messages if m.get("_msg_id") in relevant_ids]
```
但 `messages` 里**没有 `_msg_id` 字段** → `relevant` 永远为空 → `ctx_text=""` → `_summarize_context("")=""` → 不返回，`Chroma` 分支形同虚设。实际只有最后兜底（取上一条 assistant 内容）生效。

### 后果
文档描述的"Chroma 向量检索做历史关联"从未真正工作，上下文修正（`_infer_correction`）在 Chroma 路径下永远拿不到 hint。

### 修复方向（P1）
- 在 Worker 索引时把 `idx` 写回 `msg["_msg_id"]`（需在索引前 copy 一份 messages，避免污染入参）。
- 或 `run_context` 内部用 `conversation_id * 1000 + i` 重建 id 与 `relevant_ids` 匹配（需保证索引顺序一致）。

## 四、重复/冗余路由逻辑（单一来源被破坏）

### 现象
- `intent/tools.py` 的 `run_tools()` 已实现 builder→requirement 状态路由（draft/planning）。
- `core/queue.py` 第 378-380 行又写了一份 `if skill_name=="builder_agent" and proj_status in draft/planning`。
- 两份逻辑并存，且 Worker 这份才是真正生效的（见 §一）。

### 修复方向（P1）
- 路由决策**只在 `run_tools`/`pipeline` 完成**，输出到 `selected_skill`。
- Worker 删除自带状态路由，只认 `selected_skill` + `decision`。

## 五、死代码 / 文档漂移

| # | 项 | 说明 | 处理 |
|---|----|------|------|
| 1 | `INTENT_SKILL_MAP[("build","building")]` | semantic 不会产出 `building` level2，永不命中 | 删除 |
| 2 | `skills/build_agent.py` | 文档说"旧版保留兼容"，但 `skills/__init__.py` **未导入**它 → 运行时不注册（死文件）；若误导入会注册 `build_agent`+`build_agent_coder` 两个 skill，潜在命名冲突 | 删除文件，或显式保留并补 `__init__` 导入（建议删除） |
| 3 | `SkillCandidate.requires_doc` | 文档 §3.5 要求，但 `run_tools` 从未赋值/使用 | 接上：需文档的 skill（builder/requirement）标 `requires_doc=True`，Worker 校验缺文档时转 requirement_agent |
| 4 | `VALID_LEVEL1/2/INDUSTRIES` | 在 `rules.py` 与 `semantic.py` **重复定义** | 抽到 `intent/common.py` 单一来源 |
| 5 | pipeline 注释"5模块并行" | 实为"语义异步 + 4 同步模块在 LLM 等待期执行"，非真并行 | 注释修正 |

### 关于"并行内存污染"的确认（用户此前担忧）
**可排除**。各模块（`run_rules/run_context/run_safety/run_tools/run_semantic`）均为纯函数式，输入 `messages` 只读、局部变量隔离、无模块级可变共享状态；`run_semantic` 走独立 `asyncio.create_task`，其局部变量不与他人共享。**不存在读写覆盖/内存污染**。建议把"并行"表述改为"重叠执行"更准确。

## 六、其他隐患（韧性/体验）

| # | 项 | 说明 | 建议 |
|---|----|------|------|
| 1 | `classify_v2` 裸 `except` | 4 同步模块异常被静默降级为默认 dataclass；safety 异常 → 静默 risk=low | 已 `log.warning`，可接受；建议对 safety 失败显式告警，避免安全降级不被发现 |
| 2 | 双重超时 | Worker `wait_for(35)` 包 `detect_intent_v2`，内部 `classify_v2` 又 `wait_for(semantic_task, 35)` | 内超时降级返回不抛，外层不触发；冗余无害，可保留 |
| 3 | LLM 超时降级 conf=0.3 | 触发 `options` 多选项而非 sensible fallback | 评估：超时场景直接 fallback explain 可能体验更好 |
| 4 | `requires_doc` 缺失校验 | 无需求文档时仍直跑 builder | 接 §五-3 |
