## 2. 架构与数据流

```
[AI 服务 queue.py 正常路径]
  run_skill(...) ──token 流──► 累积 assistant_text
       │
       ├─ 拦截 done: 暂存
       ├─ run_safety(messages, project_constraints) → safety_risk
       ├─ run_qc(user_text, assistant_text, safety_risk)   # 并行调 3 模型
       │      └─ 聚合: 每维均值+方差, 叠加确定性地板
       ├─ publish qc 事件  ──────────────┐
       └─ publish done 事件              │
                                         │ SSE 透传(代理无需改逻辑)
[业务 proxy.py /api/chat]                 │
  publisher(): 捕获 qc 事件 → qc_result   │
  finally → _do_persist → qc_score_repo.upsert(trace_id)  ──► MySQL qc_scores
                                         │
[前端]                                    │
  chat.ts onQc → ChatView qcMap[traceId] → MessageBubble QC 徽标 + 折叠 6 维
  用户点击「⭐ 评价」→ 总体 1-10 + 展开 6 维 → POST /api/feedback(dimensions)
                                         │
[管理后台]                                │
  /admin/quality → 3 模型每维均值 + 整体每维均值 → RadarChart(4 序列)
  /admin/traces  → 列表带 qc_overall / feedback_rating
  /admin/traces/{id} → 详情: QC 表 + 用户反馈 + 对话内容 + 事件流
```

**关键结论**：业务代理的 SSE 透传机制对未知 event 类型直接转发，因此 AI 服务新增 `qc` 事件**无需改代理转发逻辑**，仅需在 `finally` 落库时捕获。`done` 仍作为流终止事件，QC 在 done 前下发，前端先看到质检徽标、再看到完成。

## 3. 六维与三裁判

- **6 维度**（顺序即雷达轴序）：`correctness 正确性 / completeness 完整性 / compliance 合规性 / efficiency 效率 / readability 可读性 / safety 安全性`，各 1-10。
- **3 裁判**：`deepseek / qwen / hy3`，并行非流式（`get_chat_model(mid, streaming=False).ainvoke`）。
- **聚合**（见 `backend/ai_service/app/qc.py`）：
  - 每维 `mean`（有效分均值）、`variance`、`scores: [deepseek, qwen, hy3]`（0=该裁判未参与/失败）。
  - `overall` = 6 维 mean 的平均。
  - `needs_review` = 任一维方差 ≥ 4（标准差 ≥ 2，裁判分歧大）或安全命中高危。
  - `partial` = 有裁判调用失败/超时。
- **混合判定（降本）**：`compliance / safety / efficiency` 叠加 `run_safety` 确定性地板（零额外成本）：
  - `safety_risk ∈ {high, critical}` → 压低 safety≤3、compliance≤4；
  - `safety_risk == medium` → safety≤6。
  - `correctness / completeness / readability` 走 LLM 三裁判（与确定性判定正交，互补）。

**韧性**：单裁判失败不影响其他；全部失败 → `partial=True` 降级；QC 整体包在 `try/except`，失败仅告警、绝不阻塞 `done`。AI 侧 `asyncio.wait_for(..., 60s)` 兜底超时。

## 4. 数据模型变更

### 4.1 新增 `QcScore` 表（`backend/business/app/models.py`）
```text
qc_scores:
  id, trace_id(idx), conversation_id, model_id,
  overall Float, result JSON(完整 qc 聚合), needs_review Bool,
  safety_risk Str, partial Bool, created_at
```
以 `trace_id` 串联 Message/Trace/Feedback，幂等 upsert（`QcScoreRepo.upsert`）。

### 4.2 `Feedback` 表新增 `dimensions` JSON 列
存气泡内 6 维细分（`{"correctness":int, ...}`），缺省 `None`（旧评价/仅整体评分）。

### 4.3 Schema 自动迁移（`backend/business/app/db.py`）
`init_db()` 原仅告警缺失表；现改为**自动 `create_all` 补齐缺失表**（仅新增、不删改已有表），
并在表存在时继续增量补列。→ 业务重启即自动建 `qc_scores`、补 `feedbacks.dimensions`，**不丢历史数据**。
