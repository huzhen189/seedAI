## 三、未实现但需你决策的功能（a/b/c 方案）

> 这些项文档列为 MVP/M1/M4 能力，但代码未做；需要你定**是否做、做多少、怎么做**。每条给三套方案。

### ① 配额限流（free 50 次/天 + 全局令牌桶）
- **现状**：docs/03 §4 写「free 默认 50 次/天 + 全局令牌桶」；实际仅 `record_model_usage` 计数模型用量，**无配额拦截、无全局限频**（`proxy.py` 直接放行）。
- **影响**：恶意/脚本可无限刷生成，烧模型 token；无套餐差异基础。
- **a 方案（现在就做完整版）**：建 `quota:daily:{user_id}` + `ratelimit:global:gen` 两个 Redis 计数器；`/api/chat` 入口查 quota（free 50/天超额返 `RATE_LIMITED` 帧）+ 全局令牌桶（如 20/秒）。改动小、纯 Redis，约 0.5 天。
- **b 方案（先做最小可用）**：只做 free 50/天 每日配额（最关键的防刷），全局令牌桶暂缓。约 0.25 天。
- **c 方案（暂不做）**：保持现状，等收费/多套餐时一并做（依赖 ② 的数据模型）。风险：上线即暴露无限生成。

### ② RAG 注入（components/memory 增强生成）
- **现状**：docs/04 §9/§10 设计 components+memory 双 Collection；实际 `generate_site.py` **无 RAG 检索调用**，Chroma 未预置数据；`rag_retrieve` 仅作为用户显式触发的独立 Skill 存在。
- **影响**：生成质量缺「组件库/历史上下文」增强，复用度低、风格不稳定。
- **a 方案（完整接入）**：① 接 Qwen text-embedding；② 预置 `components` 集合（你提供若干优质模板片段）；③ `generate_site` Planner 前自动检索 components+memory 注入 prompt；④ 生成后异步回写 `memory`。约 2~3 天（含数据准备）。
- **b 方案（轻量接入）**：只接 `components` 检索注入（不接用户 `memory` 闭环），先提升一次性生成质量；`memory` 回写留待后续。约 1 天。
- **c 方案（暂不做）**：当前 Planner→Coder→Reviewer 已能产出可用站点，RAG 作为质量增强后置到 M2。

### ③ Trace / Feedback / UsageLog 数据模型 + 回放 + AI 质量指标 + 1-10 评分
- **现状**：docs/03 §7/§8/§10 设计 `Trace`/`TraceEvent`/`Feedback`/`UsageLog`/`Artifact` 五张表；实际**均未建**。后果：管理页「回放」无数据、`/admin/traces/:traceId` 为 TODO、6+1「AI 质量」维度无数据源、前端 👍/👎 静默 no-op（后端 `/api/feedback` 未实现）。
- **影响**：运营无法复盘单次生成、无法量化模型质量、无法收集回归数据集；管理后台深度受限。
- **a 方案（一次性建全）**：建 `Trace`/`TraceEvent`/`Feedback`/`UsageLog`（含 `Message.trace_id` 关联）；SSE 生命周期内写 `TraceEvent`；`/api/feedback` 落 `Feedback`；管理页加「回放」+「AI 质量」标签页（聚合 Trace/UsageLog/Feedback）；前端评价升级为 1-10 分 + 评论。约 3~4 天（后端表+写入+聚合，前端两页）。
- **b 方案（先做反馈 + 质量指标，回放暂缓）**：建 `Feedback`+`UsageLog`，落 👍/👎 与 1-10 评分、成本账本；管理页加「AI 质量」卡片；`TraceEvent` 回放留待后续。约 2 天。
- **c 方案（暂不做）**：保持基础运行指标看板，回放/质量/反馈全后置。风险：模型迭代无量化依据。

### ④ WebLLM 浏览器内首过
- **现状**：docs/02 §6 设计 `@mlc-ai/web-llm` 本地首过；实际前端**无 WebLLM 选项**，全部走云端 SSE。
- **影响**：首字延迟高、云端 token 成本高、无隐私/离线兜底。
- **a 方案（完整接入）**：引入 `@mlc-ai/web-llm`（~2-5GB 权重 CDN 下载 + Cache Storage 预取），`useWebLLM` 封装，`ModelSelector` 加「本地 WebLLM」选项，同构事件帧，失败回退云端。约 3~5 天（含 WebGPU 兼容处理 + 模型选型调优）。
- **b 方案（轻量试点）**：只接「本地 WebLLM 规划（Planner）」、Coder 仍走云端（降低权重体积与质量风险），验证首过链路。约 2 天。
- **c 方案（暂不做）**：云端 Orchestrator 已够用，WebLLM 作为 M3 性能优化后置。

### ⑤ 分享链接（只读分享）
- **现状**：docs/01 §2.1 MVP 列「分享链接」；实际 `Project` 无 `share_id` 字段、无分享端点、无只读分享页。
- **影响**：用户无法把生成的站点分享给他人（只能自己看 COS 直链）。
- **a 方案（完整分享）**：`Project` 加 `share_id`（UUID，可空）；`POST /api/projects/{id}/share` 生成、`DELETE` 取消；前端 `/share/:shareId` 只读页渲染该会话最新产物（iframe COS 直链，不暴露登录态）。约 1~1.5 天。
- **b 方案（最小分享）**：不加独立分享页，直接暴露「复制预览链接」按钮（COS 直链 + 是否公开开关）。约 0.5 天。
- **c 方案（暂不做）**：个人自用阶段不需要分享，后置到 M2。

### ⑥ 控制面 Orchestrator 真实实现（启停 / 扩缩容）
- **现状**：docs/03 §9 / docs/02 §10 设计 `DockerComposeOrchestrator`；实际 `/admin/scale`、`/admin/stop` **仅返回 ack stub**，不执行任何操作。
- **影响**：超管在后台点「停止/扩缩容」无效，纯占位。
- **a 方案（接 DockerComposeOrchestrator）**：后端实现 `stop(service)`/`scale(service, replicas)` 封装 `docker compose`（或 `docker` CLI），超管执行 + 二次确认；前端控制面接真实结果。约 1~1.5 天（含权限校验 + 安全）。
- **b 方案（占位保留，补执行日志）**：stub 改为真正调用本地 `docker` 命令但仅记录、不强制；或等部署形态确定（Docker Compose vs K8s）再做。约 0.5 天。
- **c 方案（暂不做）**：当前单机自托管，手动启停即可，控制面保持占位。
