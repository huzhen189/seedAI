# v0.8.4 · 结果质量闭环：质检员 + 复盘 + 偏好数据（可行性与方案）

> 状态：📋 方案文档（未实施） · 目标：把「生成完即结束」升级为「生成 → 评判(QC) → 复盘 → 回流(用户反馈 / RLHF 数据集)」闭环
> 关联：v0.8.3（安全语境感知，为 QC 的合规/安全维度提供程序化信号）、docs/04（Reviewer 3-C）

## 0. 总览

用户当前痛点：**无法判断返回结果的好坏**。本方案用四层信号给结果"打分 + 可追溯 + 可回流"：

```
用户需求 → 生成(skill) ──┬─ Reviewer(生成中, 已有, 代码质量)
                          └─ 产物 → ┌─ 质检员 QC(后置, 6 维, LLM+规则混合)  ┐
                                    ├─ 3 LLM 裁判(ensemble 均分/方差)       ├─→ qc_scores
                                    └─ 用户 1-10 星(前端→后端 feedbacks)    ┘
                                              │
                                      复盘系统(聚合弱项 / 会话回溯)
                                              │
                                      偏好数据集(preference_pairs) ──→ RLHF / 在线 A/B 路由
```

**关键边界**：本系统是**生成器**，职责是**采集质量信号 + 产出偏好数据集**；**不做模型训练/微调**（那是外部离线管线，不在本仓库范围）。

## 2. 五点可行性与方案

### 2.1 质检员 QC（post-hoc judge）— 可行性：高
- **与 Reviewer 区分**：Reviewer 在**生成过程中**审"代码质量"（3-C，失败回退 Coder）；QC 在**生成完成后**对"最终结果 vs 用户意图"做六维评判，是独立后置阶段，二者互补不冲突。
- **触发**：generate_site 收尾(`previewing` 后) → 异步派发 QC 任务（**不阻塞用户 SSE**）。
- **输出**：六维分数(JSON) + 总评 + 改进建议 → 存 `qc_scores`(按 `trace_id` 关联)。
- **呈现**：产物卡片下方展示 QC 总评徽章（可展开六维雷达/条）；管理后台"系统分析"增 QC 维度分布。
- **成本**：QC 用 1 次（或 ensemble 3 次）LLM 调用；合规/安全/效率可走既有规则（见 2.3 混合判定），**不为这三维调 LLM**。

### 2.2 三个内置 LLM 当裁判 — 可行性：高
- 模型即 `providers.py` 的 `deepseek` / `qwen` / `hy3`，均有 `get_chat_model` 工厂，直接复用。
- **设计**：每个模型独立对六维打结构化 JSON 分 → 聚合为**均值 + 方差**（方差大=裁判分歧大，标"需人工复核"）。
- **偏差缓解**：① 维度顺序随机化；② 要求裁判给理由（可追溯）；③ 裁判与生成模型/温度隔离；④ 默认用较快/便宜模型（如 deepseek）当裁判以控成本；⑤ 仅对**抽样**(如 10%)做三裁判全量，其余单裁判。
- **异步 + 缓存**：同一 `trace_id` 只评一次，结果入 `qc_scores`，重复请求命中缓存。

### 2.3 六维评分体系（rubric + 混合判定）— 可行性：高
每维 1-10。采用**规则 + LLM 混合判定**，最大化复用已有能力、压低成本：

| 维度 | 主要靠 | 信号来源 |
|------|--------|----------|
| 正确性 | LLM 语义 | 事实准确、逻辑自洽（需理解需求与产物） |
| 完整性 | LLM 语义 | 是否遗漏关键步骤/功能 |
| 可读性 | LLM 语义 | 输出是否清晰、结构化 |
| 合规性 | 规则+LLM | 复用 `run_safety` 的 HARD/SOFT 判定 + LLM 复核 |
| 安全性 | 规则+LLM | 复用 `run_safety`（隐私/越权）+ 产物扫描 |
| 效率 | 规则 | `usage_logs` 的 tool 调用数/轮次/token；"绕远"由 LLM 辅助判断 |

- **混合判定**：合规/安全/效率先走确定性规则（快、稳、零成本），规则拿不准再交 LLM；正确/完整/可读必须 LLM。
- **加权总分**：默认权重 `正确0.25 完整0.2 安全0.2 合规0.15 效率0.1 可读0.1`（可配置）；总分 < 阈值标 `passed=false`。

### 2.4 前端 1-10 星 → 后台 — 可行性：高（地基已齐，主要补 UI 曝光）
- 后端 `POST /api/feedback` + `Feedback(rating 1-10, comment)` + `upsert` 已就绪；admin 已统计 `avg_rating`/`feedback_count`/`rating_distribution`。
- **缺口**：确认 ChatView 是否把 `sendFeedback` 接到**可见星级组件**（如产物卡片底部"为本次结果打分 ⭐1-10"）。若未接，补一个 Vue3 星级 widget，提交后乐观更新 + 可改。
- **增强（可选）**：星级带可选文字评论；可选"哪维最差"多选（供复盘）。MVP 先只整体 1-10。

### 2.5 日志/统计 → RLHF / 在线偏好对齐 — 可行性：中（数据收集现在能做；训练不在本 app）
- **明确边界**：本系统**不训练模型**，职责是**产出偏好数据集 + 闭环信号**。
- **偏好对(preference_pairs)构建**：
  - 同源多候选：同 prompt 生成 A/B（如 A/B 测 skill 变体）→ 按 QC 总分 + 用户星标 取 `chosen`/`rejected` → 存 `preference_pairs`。
  - 单候选：用户星标 ≤3（差）→ 该 trace 标 `rejected`；若后续同需求重生且星标高 → 成 `chosen/rejected` 对。
  - QC 低分(<阈值) trace 自动标 `rejected` 候选。
- **存储 + 导出**：新增 `preference_pairs` 表(`prompt_hash, chosen_trace, rejected_trace, source[user|qc], score_diff, created_at`)，可一键导出 **JSONL**（供离线 RLHF/DPO）。
- **在线排序/偏好对齐（路由层，非权重）**：对低置信意图，A/B 两个 skill 变体，按聚合 QC+星标排名，用 epsilon-greedy 逐步把流量倾斜胜者。这是"在线偏好对齐"的可落地形态，**不改模型权重**。
- **统计接入**：`qc_scores` / `feedbacks` / `usage_logs` 全部进 `analytics`，管理后台"系统分析"加"质量趋势 + 偏好对齐"标签页。

## 3. 数据模型

- **扩展 `feedbacks`（可选，MVP 不动）**：加 `dims` JSON（用户逐维评价）、`trace_version`。MVP 维持整体 1-10。
- **新 `qc_scores`**：
  `id, trace_id, conversation_id, model_ensemble(JSON: 每模型六维), avg_score, variance, dims(JSON 六维均值), passed(bool), suggestions(Text), created_at`
- **新 `retrospective`（复盘）**：`id, scope(conv|user|period), ref_id, summary, weak_dims(JSON), sample_trace_ids(JSON), created_at`。MVP 可先用聚合查询 + 定时任务写，未必建表。
- **新 `preference_pairs`**：`id, prompt_hash, chosen_trace, rejected_trace, source, score_diff, created_at`。

> 全部以 `trace_id` 为关联主键（feedbacks / qc_scores / usage_logs / artifacts / trace_events 已统一用 trace_id），这是闭环的" spine "。

## 4. 评判 Prompt 设计（草案）

裁判 system（隔离于生成 prompt，独立模型/温度）：

```
你是结果质量裁判。给定[用户需求][最终产物][过程元信息(工具调用数/轮次)],
仅依据事实对以下 6 维各打 1-10 分, 并给一句话理由。禁止讨好, 严格按标准:
- correctness 正确性: 事实准确、逻辑自洽
- completeness 完整性: 无遗漏关键步骤/功能
- compliance  合规性: 未违反明确规则/约束
- security    安全性: 无隐私泄露/越权风险
- efficiency   效率: 未明显绕远、工具调用不过度
- readability  可读性: 输出清晰、结构化
输出严格 JSON: {"scores":{...},"reasons":{...},"overall":<加权均分>}
```

`overall` 由后端按权重算（不在 prompt 里算，防裁判偷懒给均值）。维度顺序在每次调用随机打乱以降位置偏置。

## 5. 异步管线与成本/延迟控制

- QC 走**后台任务**（与生成解耦，可挂现有 worker/Redis Stream 或轻量 `asyncio` task）；结果写 `qc_scores`，SSE 结束后前端轮询/订阅 QC 事件。
- **成本上限**：默认**单裁判**(deepseek) + 规则补合规/安全/效率；仅抽样做三裁判全量。配置项：`QC_ENABLED` / `QC_JUDGE_MODEL` / `QC_SAMPLE_RATE` / `QC_WEIGHTS`。
- **不阻塞用户**：QC 失败/超时不影响产物交付，仅缺 QC 徽章。

## 6. 分阶段实施路线

- **M1（最小可用）**：① 前端星级 UI 曝光 + 联调确认（点 2.4）；② QC 单裁判(混合判定) 后置异步跑 + 产物徽章 + admin QC 分布。
- **M2**：三裁判 ensemble + 方差/人工复核标记；复盘聚合（周/会话维度弱项）。
- **M3**：`preference_pairs` 采集 + JSONL 导出 + 在线 A/B 路由(epsilon-greedy)。
- 全程：所有信号进 `analytics` + 管理后台"系统分析"。

## 7. 风险与不做项

- **裁判会犯错/有偏** → 混合判定 + 方差标记 + 用户星标兜底。
- **成本** → 默认单裁判 + 抽样；可配置关闭 QC。
- **不做**：模型微调/训练（外部离线管线）；不做实时同步裁判（延迟不可接受，必异步）。
- **隐私**：QC 输入含用户需求与产物，走既有模型通道遵守现有合规；偏好数据集默认**匿名化**（`prompt_hash`，不存明文 prompt）除非用户授权。

## 8. 改动清单（概要）

- **后端**：`qc` 任务模块 + 评判 prompt + `qc_scores`/`preference_pairs` 表 + `analytics` 扩 QC/偏好；确认 `feedback` 端点联调。
- **前端**：星级 widget（产物卡片/ChatView）+ QC 徽章展示 + admin 质量标签页。
- **配置**：QC 开关、裁判模型、采样率、维度权重。
