## 1. 技术栈与职责

- **框架**：FastAPI（Python）+ LangGraph + SQLAlchemy(async，若需本地表）。
- **职责**：模型抽象层（Provider 注册表）、Router+Skill 路由、generate_site 内层 LangGraph 轻量多 Agent、RAG（Chroma）、流式生成。**独享模型 API Key**。
- **数据归属**：Chroma（向量）+ Redis（队列 / 缓存）。MySQL 归业务服务（Trace/Feedback 由业务落库；本服务仅产出事件经 SSE 透传）。
- **对外**：仅对业务服务暴露。端点 `/generate`（SSE）、`/models`、`/skills`、`/tools`、`/registry`（调试）。

## 2. 模型抽象层（Provider / FallbackRouter）

目标：新增一个模型 = 写一个适配器 + 注册，前端与业务服务无感。

```python
from abc import ABC, abstractmethod
from typing import AsyncGenerator

class BaseLLMProvider(ABC):
    id: str
    label: str
    @abstractmethod
    def chat_stream(self, messages: list[dict], **kwargs) -> AsyncGenerator[str, None]: ...

PROVIDERS: dict[str, BaseLLMProvider] = {
    "deepseek": DeepSeekProvider(api_key=settings.DEEPSEEK_KEY),
    "qwen":     QwenProvider(api_key=settings.QWEN_KEY),
    "hy3":      HY3Provider(api_key=settings.HY3_KEY),
}
```

- **HY3**：OpenAI 兼容，`base_url=https://tokenhub.tencentmaas.com/v1`，model=`hy3`。
  - **关键坑（2026-07-18 修复）**：`HY3_API_KEY` 必须是腾讯 TokenHub 控制台（https://console.cloud.tencent.com/tokenhub/apikey）创建的**真正 API Key（形如 `sk-...` 或令牌串）**；文档早前记的 `TokenHub_QCSLinkedRoleInInitialization` 只是「关联角色名」，不是可用 Key，直连会被 `401002 API Key 不存在/签名校验失败` 拒绝。Demo 测试可用 `HY3_API_KEY_DEMO`（示例临时 key）。
- **扩展**：新增模型只需实现 `BaseLLMProvider` 并注册一行。
- **配置加载（2026-07-18 修正）**：`config.py` 用 `Path(__file__).resolve().parents[3] / ".env"` **绝对路径**加载项目根 `.env`，不受服务启动目录影响。此前用相对 `env_file=".env"`，本地从 `backend/ai_service/app` 启动时会找不到根 `.env` → 三个模型 Key 全空 → hy3 调 TokenHub 带空 Key 触发 401 鉴权失败。业务端 `config.py` 同样改为根绝对路径（避免回落 SQLite / 默认值）。
- **FallbackRouter（2-C）**：在 `PROVIDERS` 之上加降级路由。
  - 主模型由用户 **套餐/层级(plan/tier)** 决定（免费用便宜模型如千问，付费可用贵的如 DeepSeek/HY3）。
  - 用户前端可**手动勾选模型优先级（主 + 备）**，后端按指定顺序回退。
  - 主模型 429/超时/5xx 时按序回退；发生回退时思考事件带 `degraded: true`，前端显示「已降级」。
  - **默认降级序：HY3 → Qwen → DeepSeek**（用户指定可覆盖）。
- **端点**：
  - `POST /generate`（SSE）：内部经 Redis `queue:generate` + Worker 产出；思考事件同 §5。
  - `GET /models`：返回 `[{id, label}]` 供前端选择器。

## 3. 编排架构（Router + Skill + Hybrid）

- **外层 Router + Skill**：一个轻量 **Router**（一次小模型调用或规则）做意图识别，把请求分发到对应 **Skill**（封装好的能力单元）；Router 同时是**鉴权 + 配额闸门**（接业务服务 free 50 次/天）。
- **内层仅对 `generate_site` 保留轻量多 Agent**：`Planner → Coder → Reviewer`，且 **Planner 与 Coder 共享上下文**（不重复喂整段对话），**Designer 在 MVP 降级为 Coder 的 style 约束**（非独立 agent）。
- **其余 Skill 单次 LLM 直出**：`write_code` / `generate_doc` / `explain` / `rag_retrieve` 无需多 agent 协作。

**服务间调用关系**

```
业务服务 /api/chat (SSE) ─┬─ [WebLLM 旁路] 前端本地首过(同构事件,失败回退)
                          └─ 内网 → AI 服务 /generate (SSE, 经 Redis 队列+Worker)
                                └─ Router(意图/鉴权/配额) → 分发 Skill
                                      ├─ generate_site → {Planner→Coder→Reviewer}(LangGraph)
                                      ├─ write_code / generate_doc / explain → 单次 LLM
                                      ├─ rag_retrieve → Chroma(语义缓存)
                                      └─ 模型抽象层 + FallbackRouter → DeepSeek/Qwen/HY3
                                      └─ Redis(queue:generate / 进度 / 缓存)
```

## 4. generate_site 内层（LangGraph + Plan-and-Execute + Reflexion）

内层状态图（LangGraph）：

```python
from langgraph.graph import StateGraph, END

class SiteState(TypedDict):
    spec: str          # Planner 产出(与 Coder 共享,不重复喂原始对话)
    html: str
    reviews: int
    passed: bool

def planner(state): ...      # 需求拆解 → spec
def coder(state): ...        # spec + style 约束 → html
def reviewer(state): ...     # 静态分析 + LLM 自审 → passed / 修改建议

graph = StateGraph(SiteState)
graph.add_node("planner", planner)
graph.add_node("coder", coder)
graph.add_node("reviewer", reviewer)
graph.add_edge("planner", "coder")
graph.add_edge("coder", "reviewer")
graph.add_conditional_edges("reviewer", route_after_review,
    {"pass": END, "retry": "coder"})   # ≤3 轮,避免死循环
app = graph.compile()
```

- **演进姿态（Plan-and-Execute + Reflexion）**：
  1. Planner 产出结构化页面规格（板块 / 布局 / 风格 / 技术选型）→ 作为 `plan` 事件推前端思考面板；
  2. Coder 严格按 `plan` 生成单文件 HTML（共享 plan 上下文）；
  3. Reviewer 按计划逐条核对 + 静态分析（§12 的 3-C）；不达标进入 **Reflexion 循环**：失败原因写入短期记忆 → 反思 → 重写（≤3 次）。
- 其余 Skill 是普通函数，单次 `PROVIDERS[model_id]` 调用即返回（无状态图）。
- RAG（§9）在 `generate_site` / `generate_doc` 的 Planner/Coder 前注入组件与记忆上下文（Chroma）。
- 各调用统一走 §2 模型抽象层 + FallbackRouter。

## 5. 思考链路事件（Thinking Trace）

Router 与 generate_site 内层在协作中产出**结构化思考事件**，经业务服务转发为 SSE `think`/`node` 帧（《前端细节》§3）：

| 事件 | 触发点 | 内容示例 |
|------|--------|---------|
| node:enter router | 收到需求 | "正在识别意图、校验配额" |
| node:dispatch | Router 分发 | "判定为 generate_site,进入网站生成链路" |
| node:enter planner | 进入规划 | "拆解需求 → 页面结构/技术选型" |
| think:planner | Planner 产出 | 结构化规格(板块/布局/风格) |
| node:enter coder | 进入编码 | "开始生成单文件 HTML(含 style 约束)" |
| think:coder | Coder 推理 | 关键实现思路(可选) |
| node:enter reviewer | 进入评审 | "静态分析 + LLM 自审" |
| think:reviewer | Reviewer 结论 | 通过 / 打回原因(≤3 轮) |
| node:done | 完成 | 产物就绪 |

> 思考事件与 `token`（产物流）分离。简单 Skill 仅发 `node:dispatch` + `token`，无多余 `think` 噪音。

## 6. SkillRegistry（技能注册表）

核心是一个注册表 `SkillRegistry: dict[name → SkillEntry]`：

- `name`：技能名（如 `generate_site`）
- `intent_tags`：意图标签（辅助 Router 匹配，如 `["site","网页","页面"]`）
- `handler`：处理函数，或已编译的 LangGraph app
- `is_graph`：是否多 agent 状态图（`True` 时 `handler` 为 `graph.compile()`）
- `description`：给 Router / 前端展示的说明

**Router 分发流程**：
1. Router 用一次小模型调用（或规则）判定用户 `intent`；
2. 在 `SkillRegistry` 中按 `intent_tags` / 显式指令匹配 `SkillEntry`；
3. 调 `entry.handler(...)`，返回 `think` / `node` / `token` 事件流。

**引入新 Skill（单次 LLM 型）**：写 handler → `register_skill(name=..., intent_tags=..., handler=..., is_graph=False)` → Router 即识别分发，**无需改 Router 核心**（开闭原则）。

**引入新多 Agent**：用 LangGraph 定义状态图 → 注册 `is_graph=True` 的 Skill，`handler = graph.compile().invoke` → 对外仍表现为「一个 Skill」，Router 无感知差异。

**触发词来源**：Router 意图判定依据 **用户 prompt 语义 + 可选显式指令**（如 `/doc`、UI 按钮）；`intent_tags` 仅作语义匹配辅助，不替代模型判定。

**动态发现（M2+）**：扫描 `ai_service/skills/` 目录自动注册，支持运营 / 用户上传自定义 Skill 热插拔。

## 7. ToolRegistry（工具注册表）

**Skill vs Tool 区别**：
- **Skill**：对外暴露的「一个完整任务能力」，由 Router 按意图分发（如 `generate_site` / `explain`）；一次调用产出完整结果。
- **Tool**：Skill 内部 agent 可调用的「原子操作」，经 LLM **function calling** 暴露给模型，模型自主决定何时调用（如向量检索、COS 上传、代码执行、HTTP 请求）。

核心是一个注册表 `ToolRegistry: dict[name → ToolEntry]`：

- `name`：工具名（如 `rag_retrieve`）
- `schema`：JSON Schema（参数名 / 类型 / 描述），用于 function calling 暴露给模型
- `func`：实际执行函数
- `scope`：`internal`（仅内部 agent 调用）/ `user_exposed`（用户可见可触发，如 UI 按钮 / 显式指令）
- `risk`：`safe` / `dangerous`（危险工具需权限校验 / 沙箱隔离）

**Tool 来源分类**：
- **A. 内置工具**：代码内用 `@tool` 装饰器实现并注册（如 `rag_retrieve` / `cos_upload` / `web_search` / `code_run`）。**MVP 全部走这层**。
- **B. 运营 / 用户贡献工具（M2+）**：一个 Python 文件 + schema 声明，放入 `ai_service/tools/` 目录；启动扫描自动注册。
- **C. 第三方 MCP 工具（M3+）**：接外部 MCP server，协议自动将 MCP 能力转为 tool schema（如高德地图、GitHub）；MVP 不接。

**`rag_retrieve` 双入口**：对外是 Skill（用户说「检索我的组件库」→ Router 分发返回上下文）；对内是 Tool（`generate_site` 内部 Planner 检索上下文时调用增强质量）。一个实现、`scope=internal` 供内部、`scope=user_exposed` 供 UI / 显式指令触发。

**引入新内置 Tool**：写函数 + `@tool(name=, schema=, scope=, risk=)` → `register_tool(...)` → Skill 内部 agent 经 function calling 即可见，Router / 核心无需改动。

**关键工程约定**：
1. **重依赖懒加载**：`chromadb` / `cos-python-sdk-v5` / `playwright` 均在**函数体内** `import`；缺包时该工具返回清晰错误（`{"ok":False,"error":"..."}`），**不阻断包导入与整体注册**。
2. **导入即注册**：`tools/__init__.py` / `skills/__init__.py` 导入各模块时执行 `@tool` / `register_skill`；`registries.bootstrap()` 导入两包即完成全量注册。
3. **目录扫描热插拔（M2+）**：`bootstrap()` 额外扫描 `tools/` / `skills/` 目录，直接丢 `.py` 文件即生效；单文件异常被捕获，不阻断整体。
4. **Router 默认兜底**：`detect_intent` 无命中时兜底到 `generate_site`，保证任何输入都有响应。
5. **调试端点**：AI 服务 `GET /skills`、`GET /tools`、`GET /registry`，查看已注册能力（admin 看板可对接）。

**如何新增（开闭原则）**：
- 新内置 Tool：`tools/` 下新建模块，`@tool(name=, schema=, scope=, risk=)` 装饰函数，在 `tools/__init__.py` 加一行 import；模型经 function calling 即可见。
- 新 Skill：`skills/` 下新建模块，调 `register_skill(name=, intent_tags=, handler=, is_graph=)`；Router 自动识别分发。
- 新多 Agent：用 LangGraph 定义图，`register_skill(..., is_graph=True, handler=graph.compile())`，对外仍是一个 Skill。
- 运营/用户贡献（M2+）：直接把 `.py` 丢进 `tools/` 或 `skills/`，重启即被 `bootstrap()` 扫描注册（`risk=dangerous` 工具须走沙箱 + 权限白名单）。

**验证（本地实测）**：`bootstrap()` 实测注册 **5 Skill + 9 Tool**；`/generate` 经 Router 分发；`web_search` 无 key 走 DuckDuckGo 实返回 5 条；`rag_retrieve` / `cos_upload` 缺依赖时优雅返回错误而非崩溃。契约完全对齐 §6 / §7。

## 9. RAG 设计（Chroma，归 AI 服务）

两个用途，两个独立 Collection：

| Collection | 内容 | 何时写入 | 何时检索 |
|-----------|------|---------|---------|
| `components` | 优质组件 / 页面模板片段 + 描述 | 运营预置 / 沉淀 | 生成前：检索 Top-K 拼进 Prompt，提升质量 |
| `memory` | 用户历史项目摘要 / 对话要点 | 每次生成后异步写入 | 生成前：检索该用户相关历史，注入上下文 |

**流程**：用户需求 → Embedding → Chroma 检索（components + memory）→ 拼装增强 Prompt → 调模型 → 生成 → 异步回写 memory。

- Embedding 模型：**Qwen text-embedding**（复用 DashScope，与千问同生态）。M0 不接向量检索，M1 做 RAG/components 检索时落地。
- Chroma 初始化：`client.get_or_create_collection(name=...)`（§13）。

> **⚠ 未实装（M1 未落地）**：`generate_site` 当前**未调用** `rag_retrieve` / 不注入 components/memory 上下文——`generate_site.py` 内无 RAG 检索代码。仅 `rag_retrieve` 作为独立 Skill/Tool 存在（用户显式「检索组件库」时可触发），但未自动增强生成。Chroma 集合尚未预置数据。完整 RAG 增强待接入。

## 10. 记忆系统闭环

本系统的「记忆」不只有 Chroma，而是一个**多层读写闭环**。

**记忆分层**：

| 层 | 载体 | 内容 | 生命周期 | 读写时机 |
|----|------|------|---------|---------|
| 短期记忆 | MySQL `Trace` / `TraceEvent`（业务服务） | 本次会话完整事件流 | 一次提问；结束落库可回放 | 实时写；不进向量库 |
| 长期记忆（用户） | Chroma `memory` | 用户历史项目摘要 / 对话要点 | 跨会话、跨项目 | 生成后异步回写；生成前检索 |
| 组件记忆 | Chroma `components` | 优质组件 / 模板库 | 长期（运营沉淀） | 运营预置；生成前检索注入 |
| 语义缓存 | Chroma `cache:gen` | 相似 prompt → 历史产物（5-C） | 长期 | 生成前查（命中复用）；生成后写 |
| 反馈记忆 | MySQL `Feedback`（业务服务） | 1–10 评分 + 评论 | 长期 | 用户评分后写；既统计又回归 |

**检索闭环流程**：
```
[0] 用户需求(prompt)
 [1] 构造查询/关键词 ── A 原始 prompt(稠密 embedding) + B Planner 结构化 spec 关键词 + C 意图 + D 项目上下文(userId)
 [2] Qwen text-embedding
 [3] Chroma 检索 Top-K(混合召回): components + memory + cache:gen
 [4] 拼装增强 Prompt
 [5] Router 分发 Skill        (cache:gen 命中 ⇒ 直接复用产物,省 token)
 [6] generate_site 内层(Planner→Coder→Reviewer)
 [7] 生成产物 HTML ──→ COS 投递(preview_url)
 [8] 异步回写 memory   [9] 异步回写 cache:gen   [10] 用户评分(1–10) → Feedback(统计+回归)
 [11] 优质产物(Reviewer 通过 + 高评分)经运营审核 → 沉淀 components
 ⇒ 闭环:读前增强 → 写后沉淀 → 模型越用越准、越省 token(自我改进飞轮)
```

## 12. 优化方案（1-C / 2-C / 3-C / 4-C / 5-C）

### 12.1 生成任务并发 → 1-C：Redis 队列 + Worker 池
- 生成请求经 **Redis `queue:generate` + Worker 池**（可多副本，水平扩、重启不丢）。流程：业务 SSE 到达 → 入 `queue:generate` → Worker 消费 → Router 分发 Skill → 进度（think/node/token）经 **Redis Stream** 写入 `gen:stream:<traceId>`（可回放，`after` 断点续传）→ SSE 端点订阅转发前端 → `done` 结束。并发上限 = Worker 数。

### 12.2 模型故障转移 → 2-C：跨模型降级路由
- `FallbackRouter` 按用户优先级 + plan 默认回退（§2）；`degraded: true` 标记透出前端「已降级」。默认序 HY3 → Qwen → DeepSeek。

### 12.3 产物可运行校验 → 3-C：静态分析 + LLM 自审
- Reviewer 两步：① 静态分析（HTML 解析，检查 doctype、标签闭合、script 语法粗筛、禁止危险 API 如 `eval` / 注入外部源的 `innerHTML`）；② LLM 多轮自审（Reviewer 建议 → Coder 修订 → 再评审，≤3 轮）。不引入 Playwright 重依赖，显著降白屏率。

### 12.4 预览安全隔离 → 4-C：独立沙箱域 + 严格 CSP
- 独立子域（COS 默认域天然不同源）+ `sandbox="allow-scripts"`（不含 `allow-same-origin`）+ 严格 CSP（当前未注入，待绑定自定义域 + CDN 时回源重写 `default-src 'none'` + 白名单 CDN / 字体）。

### 12.5 生成结果缓存 → 5-C：语义相似缓存
- Qwen text-embedding 对用户 prompt 向量化 → 查 Chroma `cache:gen` → 相似度 > 阈值直接复用 / 微调上次 HTML，省 token。用户可「强制重生成」覆盖。**依赖 M1 向量库**（MVP 暂不做，或以精确 prompt 哈希过渡）。**（⚠ 未实装：cache:gen 回写未接（§10 闭环未打通），generate_site 不查缓存）**

## 13. 资源层连接池（Chroma / Redis，归 AI 服务）

- **Redis（redis.asyncio 连接池）**：`pool = redis.asyncio.ConnectionPool.from_url(REDIS_URL, max_connections=50, decode_responses=True, health_check_interval=30)`；客户端全局复用。用途：`queue:generate` / `gen:progress:<traceId>` / `gen:inflight` / `cache:models` / `cancel:<traceId>`（C1）/ `cache:gen`。
  - **协议兼容（v0.4.4 修复）**：部分云 Redis（老版本/代理）不支持 redis-py 默认的 `HELLO`(RESP3) 握手，会直接 `unknown command HELLO` 导致 `/generate` 运行时 500。`queue.py` 的 `RedisBackend` 强制 `protocol=2` 避开 `HELLO`；并在构造时做一次**真实 PING 握手**探测，连不上或不兼容即回退 `MemoryBackend`（进程内 asyncio 队列，单进程开发足够），保证本地无 Redis 也能跑通闭环。`DEV_MEMORY_QUEUE=1` 或 `REDIS_URL=memory://` 可强制内存模式。
- **Chroma（单例 HttpClient + httpx 连接池）**：复用**单例 `HttpClient`**，注入自定义 `httpx.AsyncClient(limits=httpx.Limits(max_connections=50, max_keepalive_connections=20))` 复用 TCP 连接。启动时 `client.get_or_create_collection(name=...)` 确保集合存在。
- **统一约束**：客户端均在服务启动时初始化为模块级单例，关闭时释放；禁止请求内反复新建连接。
