## 二、改进一：AI 核心流程重构（10 项决策已确认）

> 详案：`docs/AI核心流程-现状与新方案对比.md`

### 2.1 用户新方案（4 步，含 2026-07-23 更新）

1. **上下文融合**：用户输入 + 最近 5 条（Redis→DB）+ 项目关键词 + 项目记忆 + **用户偏好（Chroma user index）** → 语义判断
2. **执行计划**：多任务拆分 → 按闲聊/建站两大方向分流；不确定 → 仅高风险反问
3. **闲聊方向**：安全/项目校验 → LLM 答 → 多模型打分 → 低分轻量重答 1 轮
4. **建站方向**：安全/项目校验 → **现有 Planner→Coder→Reviewer→Reflexion 流水线（不变，加阶段标签+自评）** → 多模型打分 → **低分定位失败阶段 → 只重启该阶段（保留上游产物，≤3 轮 / 单步 30s，兼容中断续跑）** → best-of-N 保留最高分

### 2.2 已确认决策（2026-07-22/23）

**架构决策**
| # | 决策 | 含义 |
|---|------|------|
| 1 | step1 复用 5 模块意图管道 | `detect_intent_v2` 不动架构，仅扩展入参 |
| **2** | **建站保持现有 Planner→Coder→Reviewer→Reflexion** | **不拆多 agent**；加阶段标签 + 结构化自评（为自修复定位铺路） |
| 3 | 失败定位 = 阶段自评 × QC 交叉验证 | 双信号逐维比对，定位到具体阶段 |
| 4 | 单步 30s 超时 / 每阶段 ≤3 轮 | 预算封顶，超时降级 |
| 5 | 反问仅高风险操作 | 与现状 `confirm` 对齐 |
| 6 | **自修复与中断续跑共设计** | repair state 存 Redis `repair:{trace_id}`；兼容现有 Stream XRANGE/XREAD；中断后从上次 repair 轮续跑 |

**优化决策**
| # | 决策 |
|---|------|
| 1 | 保留上游、只重启失败阶段 |
| 2 | 闲聊加 1 轮轻量重答（跑题/事实错/安全软警示） |
| 3 | best-of-N 跨轮保留最高分产物 |
| 4 | 每轮轻量自评 + 仅最终全量三裁判 QC |

### 2.3 落地顺序（Phase A→D）

| Phase | 内容 | 风险 | 主改文件 |
|-------|------|------|----------|
| **A** 上下文深度融合 | 业务侧组装「最近 N 条 + 项目记忆 + 用户偏好(Chroma)」→ `detect_intent_v2` 扩展入参 | 低 | `proxy.py`, `intent/pipeline.py`, `intent/context.py` |
| **B** 建站阶段标签+自评 | 在现有 Planner/Coder/Reviewer/Reflexion 各阶段注入 `stage` 标签 + 结构化自评 JSON；可选在 Coder 后加 Style 阶段 | 低 | `skills/generate_site.py` |
| **C** 自修复闭环+中断续跑 | `core/repair.py`(新增)；repair state 存 Redis；兼容 Stream 回放/续接；局部重启 + best-of-N | 核心增量 | 新 `core/repair.py`, `core/queue.py` |
| **D** 闲聊修复+交互收敛 | 闲聊 1 轮重答 + 反问仅高风险 | 低 | `skills/explain.py`, `queue.py` |

### 2.4 自修复 + 中断续跑联合设计（Phase C 新增）

```
前端中断/刷新
  → sessionStorage trace_id + after 参数重连
  → Worker 读 repair:{trace_id} 状态
     { "round": 2, "failed_stage": "coder", "done_stages": ["planner"], "best": {...} }
  → 从断点续跑（不重头来）
  → 正在跑的 LLM 流式调用被打断 → 标记 failed → 进入下一轮 repair（预算内覆盖）
```

续跑基础设施（v0.5.0）已具备：Redis Stream XRANGE/XREAD、`after` 参数、`trace_id` 幂等。**只需要加 repair state 持久化这层**。

## 三、改进二：产品记忆压缩（四层架构）

> 详案：`docs/seedai-memory-compression-plan.md`

### 3.1 四层记忆架构（L0→L1→L2→L3，新增 L2+/L3）

| 层 | 内容 | 存储 | 生命周期 | 现状 |
|----|------|------|----------|------|
| L0 短期原始 | 完整对话 | MySQL `Message` | 永久 | ✅ |
| L1 短期压缩 | 滚动摘要 ≤200 字 | Redis `summary:{conv_id}` **1 天 TTL**（从 7 天缩短） | 1 天，过期从 MySQL 重压 | ⚠️ 需改 |
| **L2 对话精炼** | **done 后 LLM 精炼本轮完整对话 → 改写 Message.content → 再压 L1** | MySQL `Message.content`（覆写）+ Redis 摘要（刷新） | 永久 | ❌ 新增 |
| **L2+ 长期结构化** | 决策/约束/偏好/产物/事实 + `importance(1-5)` + `project_id` | MySQL `project_memories` + Chroma `project_memory`（分项目） | 永久（compaction 防膨胀） | ❌ 新增 |
| L3 用户卡片 | 行业/风格/技术栈/禁忌 + 偏好向量 | `User.card_json` + Chroma `user_preferences`（分用户） | 永久 | ❌ 新增 |

### 3.2 L1 改动（Redis TTL + 过期回退）

- **现状**：`proxy.py` `save_summary` 用 `r.setex(f"summary:{conv_id}", 86400 * 7, text[:1000])`——**7 天 TTL**。
- **目标**：`86400 * 1`（1 天）。Redis key 过期后 `get_summary` 返回 None → 触发降级：从 MySQL `Message` 取最近完整对话 → **调 LLM 重新压缩 → 写回 Redis**。

### 3.3 L2 对话精炼（done 后改写 DB content）

- **时机**：trace `done` 事件（与 QC、git 提交同一钩子点）。
- **流程**：
  1. 取本轮完整对话（user + assistant 原始流式拼接文本）
  2. 调 LLM（小模型 deepseek-chat）做**精炼重写**：去冗余废话、保留完整信息、结构清晰、语气连贯自然
  3. 用精炼后文本 **覆写 MySQL `Message.content`**（仅 assistant 侧，不碰 user）
  4. 用精炼后文本 **更新 L1 滚动摘要**（`maybe_compress_summary` 或直接 replace）
  5. 前端 SSE 的 `done` 事件里，可以把精炼后文本作为最终展示（可选项，渐进：先替换 DB，后续加 SSE 覆写事件）
- **幂等**：`Message` 加 `refined` flag 列或 Redis `refined:{msg_id}` 标记，避免重复精炼。
- **失败降级**：精炼失败 → 保留原始 content → 仅 warn，不阻断。

### 3.4 L2+ 蒸馏 → Compaction

- **蒸馏**（任务 done）：LLM 从 L2 精炼后对话 + 产物 → 抽取结构化条目 → 存 MySQL + Chroma（**带 `project_id`/`user_id` metadata**）
- **Compaction**（> 50 条或 30 天）：LLM 合并重叠/过期 → 精简集覆盖，`importance` 升序删保高值
- **检索**（新任务 step1）：Chroma top-k（按 `project_id` 过滤）+ 最近 5 条 → 拼成 context

## 四、改进三：Chroma 多方向索引设计（新增）

> Chroma 现状仅 3 个集合：`conversation_context`（分会话）、`memory`（无隔离）、`components`（公共）。需要按用途和隔离维度拆成 **6 个独立集合**。

### 4.1 六集合架构

| 集合 | 用途 | 隔离 key | 增删频率 | 典型查询 |
|------|------|----------|----------|----------|
| `conversation_context` | 对话上下文相似度检测（现有） | `conversation_id` | 每条消息写入 | `find_relevant_messages` |
| `user_preferences` | **用户喜好/禁忌/习惯** | `user_id` | 低频（每次任务结束 distill） | 「用户喜欢什么风格」→ 送 intent |
| `project_memory` | **项目规则/风格/需求/决策历史** | `project_id` | 中频（每次任务结束 distill） | 「这个项目之前决定了什么」→ 送 step1 |
| `project_code` | **大型项目临时代码语义索引**（上下文窗口有限时检索相关代码块） | `project_id` | 高频（每次生成/修改后增量 upsert） | 「找与这个需求最相关的已有代码」→ 送 Coder/Reflexion |
| `error_patterns` | **错误模式库**（全局跨项目，让 agent 避坑） | 无（全局共享） | 低频（修复确认后写入） | 「以前类似的错误是怎么修的」→ 送 子阶段修复 |
| `components` | 公共组件库（现有） | 无 | 不变 | `build_rag_context` |

### 4.2 各集合数据结构

**`user_preferences`**
```
{ user_id, type: "style"|"constraint"|"habit", content, importance, ts }
```
- 例：`{ user_id:42, type:"style", content:"用户偏好暗色主题，蓝金配色，简洁无动画", importance:5 }`

**`project_memory`**
```
{ project_id, user_id, type: "decision"|"constraint"|"requirement"|"artifact", content, importance, ts }
```
- 例：`{ project_id:7, type:"decision", content:"首页走静态 HTML 不调 API，SEO 优先", importance:5 }`

**`project_code`**
```
{ project_id, file_path, chunk_hash, content, function_name, language, ts }
```
- 增量 upsert：代码生成后按函数/区块切片，hash 去重（chunk_hash → 不覆写无变化块）
- context window 有限时检索 top-k 相关代码块，注入 Planner/Coder prompt

**`error_patterns`**（全局）
```
{ error_type, trigger_pattern, fix_pattern, success_count, project_ids[], ts }
```
- 例：`{ error_type:"flexbox overflow", trigger:"display:flex 无 min-width:0", fix:"加 min-width:0 或 overflow:hidden", success_count:12 }`
- Agent 修复成功后 upsert（success_count +1）；失败不写

### 4.3 隔离实现方式
- **metadata where 过滤**（推荐，不改 collection 名）：`where={"project_id": project_id}` / `where={"user_id": user_id}`。Chroma 原生支持 metadata 过滤，简洁有序。
- 对 `project_code` 使用 `project_id + chunk_hash` 复合 id 做幂等 upsert。

## 五、已知代码缺口（2026-07-23 审查发现）

### 5.1 QC 只评分不纠错
`qc.py` 的 `run_qc` 仅返回评分 JSON，never 修改 `assistant_text`。低分文本原封不动送给用户。
→ **修复**：对接 Phase C 自修复闭环（§2.4）。

### 5.2 Chroma memory 无隔离（跨用户污染）
`save_memory` 写 `trace_id` 不写 `project_id`/`user_id`；`build_rag_context` 检索无 `where` 过滤。
→ **修复**：§4.1 六集合架构全覆盖。

### 5.3 L1 摘要过期无降级
Redis key 过期后 `get_summary` 返回 None → 本次对话丢失历史压缩上下文。
→ **修复**：§3.2 过期回退重压。

## 七、补充优化建议（2026-07-23）

除了你提的 5 点，还有几个值得顺手做的：

### 7.1 L2 精炼结果的前端可见性
精炼后改写 DB content，但前端仍展示原始流式文本。建议在 `done` 事件后追加一个 `refined` 事件（`{"type":"refined","content":"..."}`），让 `MessageBubble` 平滑切换展示精炼版。成本：+1 个 SSE 事件，零额外 LLM 调用。

### 7.2 `project_code` 的预热时机
不要等项目大到 context window 越界了才开始索引。建议在首次 `generate_site` 的 `done` 之后**后台异步**做一次全量代码切片索引（不阻塞用户），后续修改只做增量 upsert。这避免了"第一次用到这功能时等了很久"的体验。

### 7.3 错误模式库的冷启动
`error_patterns` 初始为空时等于废的。建议先硬编码 20-50 条**常见 HTML/CSS 错误模式**（如 flexbox overflow、z-index 层叠、grid 溢出等）作为种子数据，随 Agent 运行逐渐被真实数据覆盖。

### 7.4 记忆蒸馏中增加"用户风格偏好学习"
在 L2+ 蒸馏时，不只抽取事实性条目，还可以抽取**风格偏好**：比如「用户连续 3 次要求改圆角从 8px 到 16px」「用户每次都说"不要太花哨"」→ 写入 `user_preferences`。下次生成直接按用户偏好设置默认值，减少重复修正。
