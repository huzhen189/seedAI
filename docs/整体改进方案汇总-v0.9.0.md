## 二、改进一：AI 核心流程重构（9 项决策已确认）

> 详案：`docs/AI核心流程-现状与新方案对比.md`

### 2.1 用户新方案（4 步）

1. **上下文融合**：用户输入 + 最近 5 条（Redis→DB）+ 项目关键词 + 项目记忆 + 用户卡片 → 语义判断
2. **执行计划**：多任务拆分 → 按闲聊/建站两大方向分流；不确定 → 仅高风险反问
3. **闲聊方向**：安全/项目校验 → LLM 答 → 多模型打分 → 低分轻量重答 1 轮
4. **建站方向**：安全/项目校验 → 产品/开发/样式/测试/修改 多 agent 流水线 → 多模型打分 → **低分定位失败环节 → 只重启那步（≤3 轮 / 单步 30s）** → `best-of-N` 保留最高分

### 2.2 9 项已确认决策

**5 硬约束**
| # | 决策 | 含义 |
|---|------|------|
| 1 | step1 复用 5 模块意图管道 | `detect_intent_v2` 不动架构，仅扩展入参 |
| 2 | 建站多 agent = `generate_site` 内部阶段 | 单 trace / 单产物目录 / 单 git 版本 |
| 3 | 失败定位 = agent 自评 × QC 交叉验证 | 双信号定位，逐维比对 |
| 4 | 单步 30s 超时 / 每子任务 ≤3 轮 | 预算封顶，超时降级 |
| 5 | 反问仅高风险操作 | 与现状 `confirm` 对齐 |

**4 优化方向**
| # | 决策 |
|---|------|
| 1 | 保留上游、只重启失败子 agent |
| 2 | 闲聊加 1 轮轻量重答（跑题/事实错/安全软警示） |
| 3 | `best-of-N` 跨轮保留最高分产物 |
| 4 | 每轮轻量 agent 自评 + 仅最终全量三裁判 QC |

### 2.3 落地顺序（Phase A→D）

| Phase | 内容 | 风险 | 主改文件 |
|-------|------|------|----------|
| **A** 上下文深度融合 | 业务侧组装「最近 N 条 + 项目记忆 + 用户卡片」→ `detect_intent_v2` 扩展入参 | 低 | `proxy.py`, `intent/pipeline.py`, `intent/context.py` |
| **B** 建站多 agent | `generate_site` 重构为 product/code/style/test/modify 阶段 | 中 | `skills/generate_site.py`(重构), `queue.py`(事件) |
| **C** 自修复闭环 | `core/repair.py`(新增) → 失败定位 + 局部重启 + best-of-N | 核心增量 | 新 `core/repair.py`, `core/queue.py` |
| **D** 闲聊修复+交互收敛 | 闲聊 1 轮重答 + 反问仅高风险 | 低 | `skills/explain.py`, `queue.py` |

## 三、改进二：产品记忆压缩（短期 → 长期结构化）

> 详案：`docs/seedai-memory-compression-plan.md`

### 3.1 背景

用户新流程 step1 需要「最近 5 条 + 项目记忆 + 用户卡片」— 现状有对话摘要（Redis 7 天）和 Chroma 原始消息向量（从不蒸馏），但缺**结构化长期项目记忆**，且 Chroma 记忆检索**无项目/用户隔离**（跨用户污染）。

### 3.2 三层架构

| 层 | 内容 | 存储 | 现状 |
|----|------|------|------|
| L0 短期原始 | 完整对话 | MySQL `Message` | ✅ 已有 |
| L1 短期压缩 | 滚动摘要 ≤200 字 | Redis `summary:{conv_id}` 7 天 TTL | ✅ 已有 `maybe_compress_summary` |
| **L2 长期结构化** | 决策/约束/偏好/产物/事实 + `importance(1-5)` | MySQL `project_memories` + Chroma `memory:{project_id}` | ❌ 新增 |
| L3 用户卡片 | 行业/风格/技术栈/禁忌 | `User.card_json` 或独立 `user_cards` | ❌ 新增 |

### 3.3 蒸馏 → Compaction

- **蒸馏**（任务 done）：LLM 从 L1 摘要 + 本轮消息 + 产物 → 抽取结构化条目 → 存 MySQL + Chroma
- **Compaction**（> 50 条或 30 天）：LLM 合并重叠/过期 → 精简集覆盖，`importance` 升序删保高值
- **检索**（新任务 step1）：Chroma top-k（按 `project_id` 过滤）+ 最近 5 条 → 拼成 context

## 四、代码审查发现的额外缺口（2026-07-23）

### 4.1 QC 只评分不纠错

`qc.py` 的 `run_qc` 仅接收 `assistant_text` 原封打分，**返回值是 `{overall:7.2, needs_review:false}` 结构，never 修改 `assistant_text`**。`queue.py` 里 `qc` 事件在 `done` 前发布，但消息内容一字未改。

→ **后果**：即使 QC 标 `needs_review=true` 或 `overall<4`，用户收到的仍是低质量原文，前端只展示评分卡片。
→ **修复**：对接改进一 Phase C（自修复闭环）—— QC 低分 → 定位失败环节 → 局部重启。

### 4.2 Chroma 无项目/用户隔离（跨用户污染）

`knowledge/chroma.py` 实测：

| 集合 | 写入 metadata | 检索 where 过滤 | 问题 |
|------|--------------|-----------------|------|
| `conversation_context` | `{"conversation_id":..., "msg_id":...}` | `where={"conversation_id":...}` ✅ | conversation 间接关联 project，但 Chroma 内无 `project_id` |
| **`memory`** | `{"trace_id":..., "tags":...}` | **无任何 where** ❌ | **用户 A 的记忆会被用户 B 检索到，跨用户污染** |
| `components` | 公共组件 | 无 | 无所谓 |

`build_rag_context` 检索 memory 的代码：

```python
mems = retrieve(query, settings.chroma_collection_memory)  # ← 无 project_id/user_id 过滤
```

→ **后果**：不同用户的建站风格/偏好混淆，语义检索可能返回无关用户的记忆。
→ **修复**：`save_memory` 加 `project_id`+`user_id` metadata；`retrieve` / `build_rag_context` 加 `where` 过滤；或用 `memory_{project_id}` 分集合。

## 五、实施优先级建议

| 优先级 | 改动 | 理由 |
|--------|------|------|
| **🔴 P0 立即** | Chroma 加项目/用户隔离 | 安全补丁，跨用户污染，改动小、独立 |
