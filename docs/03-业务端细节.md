# AI 生成器平台 — 业务端细节（Business API）

> 本文档定义业务服务（FastAPI / Python）的实现细节：**鉴权与用户系统、RBAC 三级、配额限流、SSE 代理、缓存与异步持久化、对话追踪回放、运营指标、管理后台、数据模型、连接池、Schema 自动初始化**。
> 跨服务架构见《项目总览》；前端契约见《前端细节》；AI 生成链路见《AI 核心端细节》。

## 1. 技术栈与职责

- **框架**：FastAPI（Python）+ SQLAlchemy(async) + Pydantic。
- **职责**：唯一对外入口。鉴权 / JWT、用户、项目、消息、分享、配额限流、用量计量、对话 SSE 代理、缓存读写、错误队列对账、指标聚合、管理后台。
- **不持有模型 API Key**：生成类请求转发给 AI 服务（内网）。
- **数据归属**：MySQL（业务数据）+ Redis（会话 / 限流 / 缓存 / 队列对账）。Chroma 归 AI 服务。

> **配置加载（重要）**：`config.py` 用 `Path(__file__).resolve().parents[3]/.env` **绝对路径**加载项目根 `.env`，不受启动目录影响（修复本地直跑时相对 `env_file=".env"` 找不到根 `.env`、回退 SQLite 的坑）。`.env` 里的 `MYSQL_URL` 常用同步驱动 `mysql+pymysql`，但业务端是异步 SQLAlchemy，**`config.model_post_init` 会自动把 MySQL URL 升级为异步驱动 `mysql+aiomysql`**（与 docker-compose 注入的 `DATABASE_URL=mysql+aiomysql` 一致），本地直跑也能连同一套 MySQL。
> **启动硬约束**：`main.on_startup` 调 `init_db()` 会连 MySQL 建表，因此**业务服务启动即要求统一 MySQL 可达**。DB 没起来服务会启动失败（fail-fast），不会悄悄回落 SQLite。本地/测试/生产共用同一套 `.env`（DB、云存储 COS、三大模型 Key 全统一），不另建本地库。
>
> **本地免 docker 开发**：`.env` 的 `MYSQL_URL/REDIS_URL/CHROMA_URL/AI_SERVICE_URL` 已指向统一云地址（与 `.env.production` 一致），所以本机直接 `uvicorn` 跑即可连云库，**无需 docker**。一键启动脚本：`scripts/start-local.sh`（按端口强杀旧进程 → 清 pycache → **仅起业务 7101**；AI 服务(7102)由用户**自己用 cmd 启动**看日志，脚本不再自动拉起）。两个 venv 分别是 `default`/ `seedai-biz`。重启若仍报旧行为，先确认 7101/7102 上没有被旧进程占着。

## 2. 鉴权与用户系统

### 2.1 认证方式
- **密码**：服务端用 `argon2` / `bcrypt` 哈希存储（绝不存明文），使用 `passlib` 或 `argon2-cffi`。
- **短信验证码**：经 `SMSService` 抽象层；验证码存 Redis（5 分钟 + 限频）。MVP 仅 Mock。
- **会话令牌**：**JWT(Access Token 短期) + Refresh Token 存 Redis**（可吊销）。无状态 Access Token 利于横向扩容。
  - **Access Token 15 分钟**，通过 **HttpOnly + Secure + SameSite Cookie** 下发（防 XSS 盗 token）。**实现**：`auth.py` 用 `Response.set_cookie(... HttpOnly, SameSite=lax, Secure=COOKIE_SECURE)` 下发；`security.get_current_user` 优先读 Cookie（兼容 Bearer 便于调试）。
  - **Refresh Token 存 Redis**（`auth:refresh:<userId>`，可吊销，7 天）。前端请求带 `credentials`，不直接持有 token。
  - **登出**：`POST /auth/logout` 清除 Cookie（Max-Age=0），前端零存储。

### 2.2 关键流程（MVP：账号密码）
- 注册：提交 username/email + password（+ 可选 nickname）→ 哈希入库（默认 plan=`free`，nickname 缺省取 username）→ 下发 Cookie。**已落地**：`POST /auth/register`、`POST /auth/login`、`POST /auth/refresh`、`GET /auth/me`、`PATCH /auth/me`；响应体（`UserResp`）含 `nickname` 字段，令牌在 HttpOnly Cookie。
  - **`PATCH /auth/me`**：修改当前用户信息，字段可选——`nickname` / `email`（唯一，改时查重）/ `old_password` + `new_password`（改密码须验旧密码）。成功后更新 Redis 用户缓存，`GET /auth/me` 立即生效。
- `User` 模型含 `nickname`（`String(64)`，默认空）、`email`（可空，唯一索引允许多 NULL）；`init_db` 建表后**自动 diff 补齐缺失列**（v0.4.3 起）：对比 model 与数据库真实表，缺列则按 model 字段类型执行 `ALTER TABLE ADD COLUMN`（MySQL/SQLite 通用，类型由 `Column.type.compile(dialect)` 编译），只补列、不删列/不改类型。改了 model 加字段后重启即自动补齐数据库，无需手写迁移（生产仍建议 Alembic）。
- 登录：校验 password_hash → 下发 Cookie。
- 生成请求：前端带登录 Cookie → 业务服务校验（`get_current_user`）→ 透传 modelId + 上下文给 AI 服务（内网）→ 流式返回。
- 限流：登录 / 注册 / 生成接口用 Redis 令牌桶限频，防 Key 盗刷；**free plan 每日生成配额默认 50 次/天**，超额提示升级（付费预留）。**（⚠ 未实装：当前仅 `record_model_usage` 计数模型用量，无 50/天 配额拦截、无全局令牌桶限频；见 §5 步骤 2）**

### 2.3 角色与初始化
- 普通注册一律得到 `user`；**初始 `super_admin` 通过种子脚本 / 环境变量 `SEED_SUPER_ADMIN=<username>` 创建**（首次启动注入，避免角色自举漏洞）。
- 仅 `super_admin` 可在用户管理接口将 `user` ↔ `admin` 互相调整；`super_admin` 角色本身不可被降级 / 移除（防锁死）。

### 2.4 抽象层（SMSService，预留）

SMS_PROVIDERS: dict[str, SMSService] = {

## 3. RBAC 三级权限

`User.role` 取值 `super_admin` / `admin` / `user`（默认 `user`）。

| 能力 | super_admin | admin | user |
|------|:---:|:---:|:---:|
| 前台对话 / 生成 / 个人项目 | ✅ | ✅ | ✅ |
| 管理页只读（指标 / 统计 / 日志 / 思考链路） | ✅ | ✅ | ❌ |
| 控制面执行（手动启停 / 扩缩容） | ✅ | ❌ | ❌ |
| 用户与角色管理（分配 role / 改 plan / 改 quota） | ✅ | ❌ | ❌ |

- 鉴权依赖按 role 分层校验：`require_admin`（super_admin **或** admin，开放只读后台）、`require_super_admin`（仅 super_admin，开放控制面与用户管理）。
- 管理页作为 Vue 内 `/admin` 路由，与用户前台共享登录态、彼此隔离；admin 进入后控制面板置灰 / 隐藏，仅 super_admin 可见可执行。

## 4. 配额与限流

- **次数配额**：复用 plan/tier 每日生成次数（Redis 计数，free 默认 50/天）。
- **全局限频**：令牌桶（如全局 20 生成/秒），防突发压垮上游与 Worker。
- **成本账本（D1）**：按 `user_id × provider × model` 记录 token 消耗（入 `UsageLog`），供未来计费、异常用量告警、模型成本对比。
- Redis Keys：`ratelimit:<userId>`、`ratelimit:global:gen`、`quota:daily:{user_id}`、`auth:refresh:<userId>`。

## 5. 对话入口 /api/chat（SSE 代理 + 级联取消）

- **入站**：前端 `GET /api/chat`（SSE，带 HttpOnly Cookie）。**已落地**：`chat()` 手动解析登录态（`_resolve_user`），未登录**不下发 JSON 401**，而是返回一条 SSE `error` 事件（`code=AUTH_REQUIRED`、`message="Missing authentication"`）——因浏览器 `EventSource` 读不到 HTTP 401 状态码，只能解析 SSE 帧；前端据此**主动弹出登录框**（见前端 §13 鉴权失败处理）。用量按真实 `user.id` 计（`record_model_usage`）。
- **错误帧约定（已落地，v0.2.2）**：`publisher()` 捕获上游 `httpx` 连接/读取异常与 `resp.status_code>=400`，统一翻译成标准 SSE `error` 帧——`code=UPSTREAM_ERROR`（AI 服务不可用 / 上游 5xx，如 Redis 未起导致 `/generate` 入队失败）、`RATE_LIMITED`（429），`message` 取上游 `detail` 或通用中文文案，**不再裸 `yield` 原始错误字节**（此前会把 `Internal Server Error` 等裸文本透出，浏览器 EventSource 无法识别）。因浏览器对 5xx 的 SSE 响应读不到正文，此设计让前端能区分「AI 服务不可用」与「网络中断」，给出明确提示（而非笼统“连接中断”）。
- **业务侧处理**：
  1. 校验 JWT（`_resolve_user` 从 Cookie 取，无效/过期→SSE auth error）→ 取 user/role/plan；
  2. 配额检查（free 50 次/天，Redis 令牌桶）→ 超额则 `error` 帧；**（⚠ 未实装：当前跳过配额/限频，直接放行）**
  3. 建 `trace_id`（= 本次 SSE 会话），写结构化日志（带 `trace_id`）；
  4. 任务入 Redis 队列 `queue:generate`，返回 SSE 流；
  5. 订阅 AI 服务进度流：AI 服务把 think/node/token/done/error 等事件写入 Redis Stream `gen:stream:<traceId>`（可回放）；业务按 `after`(断点) `XRANGE` 回放 + `XREAD` 续接实时，逐帧转发前端。前端传 `after` 可增量续传、留空则全量回放（重连场景）。
- **业务 → AI 传输**：业务 `httpx` 流式读 AI 服务 `/generate` SSE，原样透传（全链路 SSE，无 WebSocket，见《项目总览》§5）。
- **离线继续 + 重连（断点续传）**：AI Worker 独立常驻后台，进度写入**可回放 Redis Stream** `gen:stream:<trace_id>`（见 §14）；同一 `trace_id` 再次访问 `/generate`（前端刷新 / 重连时复用同一 `trace_id`）即命中"续接"——Worker 不重新生成，仅把 Redis Stream 从断点 `after`(留空则全量)回放 + 续接实时进度透传前端。因此用户离线 / 刷新页面后，后台继续跑未完成的生成，重连即可恢复完整流程（前端用 `sessionStorage` 记录 active `trace_id`，挂载 / 切换会话时自动重连）。
- **级联取消（C1）**：前端 `AbortController.abort()` 断开 → 业务感知连接关闭 → 经 Redis PubSub 广播 `cancel:<traceId>` → AI Worker 中断当前生成。

## 6. 缓存与异步持久化（Cache-Aside + Write-Behind）

目标：活跃用户高频读取不每次打 MySQL；写入解耦、失败可重试，保证最终一致。

> **⚠ 未实装**：当前 SSE 结束由 `proxy._persist_conversation` **同步**落库（按 `trace_id` 幂等），并未走 `queue:write` 异步写队列；`reconciler.py` 仅骨架占位（`_retry_one` 直接返回成功，未真回写 MySQL），无 `queue:error` 生产者。缓存读路径（`cache:user:*`）仅登录态局部使用。完整 Cache-Aside + Write-Behind + 对账待 M2。

**读（Cache-Aside / 懒加载）**
- 活跃用户常用数据（用户资料、plan/quota、最近项目列表、用量计数）以 `cache:user:<id>:*` 存 Redis，**TTL 30 分钟**，用户有活动则续期。
- 读：先查 Redis，命中即返；未命中回源 MySQL，写入 Redis 后返回。

**写（Write-Behind / 异步）**
- 写：先**同步更新 Redis**（保证后续读一致），再把写操作投入**异步写队列**（Redis），由 Worker 落 MySQL。
- 失败：Worker 落库失败 → 进**错误队列**（Redis），附重试次数与错误原因。

**定时对账（Reconciler）**
- 定时检查器（调度任务）周期性扫描错误队列，按退避策略重试；超过最大重试次数 → 死信（dead-letter）+ 告警，交人工介入，不阻塞主流程。
- 取舍：同步写 = 注册 / 改密 / 订阅付费（用户强一致）；异步写 = 高频追加 Message / UsageLog / Artifact / memory 回写（经 `queue:write`）。写成功后对 Redis 取「更新」回填 `cache:user:*`。重试策略：指数退避，最多 5 次，超限进 `queue:error:dlq` + 告警。

## 7. 对话追踪与回放（Trace / TraceEvent / Feedback）

一次 SSE 会话 = 一个 `Trace`（`trace_id`）；会话内所有事件按 `seq` 追加到 `TraceEvent`，形成可重放时序流。`Message` 存对话气泡（user / assistant 文本），`TraceEvent` 存颗粒度更细的 SSE 事件流（think / node / token 进度 / 操作），二者通过 `trace_id` 关联。

> **⚠ 未实装**：当前 `Message` 表含 `trace_id`（用于幂等落库与重连），但 `Trace` / `TraceEvent` / `Feedback` 三张表**均未建**。回放（管理页「回放」）与反馈统计（§8）依赖它们，故均不可用；前端「回放」入口为 TODO、后端 `/api/feedback` 未实现（前端 👍/👎 为静默 no-op）。

**存储方案**：采用 **MySQL 双表 + Feedback**（方案1）。理由：已有 MySQL，SQL 直接回放 + 与 §8 指标同源计算，零额外组件；TraceEvent 只追加不改，写入复用 §6 异步写队列。

**回放视图**：管理页（super_admin 或本人）按 `trace_id` 拉 `TraceEvent` 序列 → 还原「用户说 / AI 想 / AI 做 / 操作」全过程；前端复用对话页组件渲染。普通 `user` 仅能回放自己会话（RBAC）。

**写入时机**：业务服务在 SSE 生命周期内把每个 SSE 帧（user_msg / think / node / token / done / error）与操作事件（rollback / feedback / edit）以 `TraceEvent` 追加；会话结束写 `Trace` 汇总（耗时 / tokens / 状态 / cancelled / fallback_used）。

**边界**：
- `token` 事件量大；MVP 全量落库（单文件 HTML 产物小，可接受），但 `TraceEvent.payload` 对 `token` 可仅存采样 / 增量或指向 `Artifact` 完整产物，避免爆表。
- 冷数据：量极大时热数据留 MySQL、冷数据按 retention（如 90 天）归档对象存储。
- 隐私：`Trace` 含用户原始 prompt，仅本人 / super_admin 可见；MVP 先靠 RBAC 控访问，后续可加列脱敏。

## 8. 多维度运营指标

指标由实时 Redis 计数 + 落库聚合（Trace / UsageLog / Feedback）双源计算，在 `/admin` 仪表盘「AI 质量」标签页呈现。

> **🆕 全维度统计已实装（v0.8.x）**：在 §8 的 6+1 指标之外，新增「业务接口调用 / AI 核心多意图编排 / 前端页面访问 / 前端点击热点」四块实时统计（见 §18），覆盖后端核心 + 业务接口 + 前端，满足「新增功能必接统计」强制约定；`/admin` 系统分析标签页已呈现。

> **⚠ 部分实装**：当前 `/admin/metrics`（`metrics.snapshot()`）仅提供**基础运行指标**——uptime / 请求总数 / 错误数 / 每分钟请求 / 模型用量。6+1 维度（效率 / 准确性 / 消耗量 / 回滚率 / 用户回馈 / 健康度 + RAG 检索质量）需 `Trace` / `UsageLog` / `Feedback` 聚合，表未建故**未实现**；「AI 质量」标签页未做。

**指标维度总表**

| 维度 | 指标 | 定义 / 计算 | 数据源 |
|------|------|------------|--------|
| **AI 效率** | 首字延迟 TTFT | 收到提问 → 首个 token 毫秒数（`Trace.first_token_ms`） | Trace |
| | 总生成耗时 | 提问 → done 毫秒数（`Trace.total_ms`） | Trace |
| | 吞吐 | `tokens_out / (total_ms/1000)` | Trace |

| | 迭代深度 | 单 Project 平均 `Artifact.version` 数 | Artifact |

| | 回滚率 | `count(TraceEvent.type='rollback') / count(Artifact)` | TraceEvent + Artifact |

| | 估算成本 | `Σ tokens × 单价`（¥，按模型） | UsageLog + Trace.cost |

| | 回滚到旧版占比 | 回滚目标为更早版本占比 | TraceEvent.payload |
| **用户回馈** | 点赞 / 点踩 | `count(Feedback.thumb=up/down)` | Feedback |

| | 评论数 | `count(Feedback.comment 非空)` | Feedback |
| **健康度** | 成功率 | `status='done'` / 总 Trace | Trace |
| | 取消率 | `cancelled=true` / 总 Trace | Trace |
| | 降级率 | `fallback_used=true` / 总 Trace | Trace |
| | 模型分布 | 各 model_id 的 Trace 占比 | Trace |
| | 时间 / 用户聚合 | 按 user / project / 日 / 周 / 月 切片 | 以上 |
| **RAG 检索质量** | 检索命中率 | 返回结果最高相似度 ≥ 阈值（默认 0.7）的 rag_retrieve 次数 / 总次数 | TraceEvent(rag_retrieve) |
| | 无结果率 | 返回空或全低于阈值次数 / 总次数 | TraceEvent(rag_retrieve) |

- **准确性为何近似**：AI 生成无客观真值，用三方交叉近似 —— ① Reviewer 自动化通过率；② 回滚率（用户用脚投票）；③ 显式评分（用户主动打分）。三者同向恶化即预警。

## 11. Schema 自动初始化（启动时自检建表）

- 用户尚未手动建表；要求**系统运行时一次性自检**：无表则用代码构建。
- **机制（幂等，三库分别处理）**：
  1. **MySQL**：业务服务 `lifespan` 启动事件调用 `init_db()`，`create_all` 幂等建表（只建不存在的表）后，逐表 `inspect` 对比 model 列与真实表列，**缺失列按 model 类型自动 `ALTER TABLE ADD COLUMN`**（v0.4.3 新增，schema diff，见下方）；可重复安全执行。
  2. **Chroma**：`client.get_or_create_collection(name=...)`（存在则复用）。
  3. **Redis**：无表概念（KV 即用）；启动时仅 `await redis.ping()` 探活，失败告警。
- **取舍**：MVP 用代码建表（零迁移工具、随代码演进）；生产可选 Alembic 迁移（不在 MVP）。代码建表 + 自动 diff 补列已覆盖「从零初始化 + 增量加列」两类场景（v0.4.3）；列类型变更 / 删列 / 重命名等结构性演进仍建议 Alembic 迁移。
- **落点**：`business/app/db.py` 的 `init_db()` 负责建表 + schema diff 补列；`Base` 定义于 `models.py`。AI 服务若需本地表同样在自身 `lifespan` 初始化。
- **schema diff 细节（v0.4.3）**：`_add_missing_columns(conn)` 用 `inspect` 取真实列名做差集，缺失列由 `Column.type.compile(dialect=engine.dialect)` 编译为目标库 DDL 类型串（如 `String(64)`→`VARCHAR(64)`），自动加 `DEFAULT`（有 `server_default`/标量 `default` 时）或 `NOT NULL`。**约束**：新增 NOT NULL 且无默认值的列，在表已有数据时会在 MySQL 报错，故加非空列务必带默认值。该机制只增列，不动既有列，避免误伤数据；列类型变更/删列/重命名仍需 Alembic。

## 12. 连接池管理（MySQL / Redis）

- **MySQL（SQLAlchemy async 连接池）**：
  - `create_async_engine(MYSQL_URL, pool_size=10, max_overflow=20, pool_pre_ping=True, pool_recycle=1800)`。
  - `pool_pre_ping=True` 取连接前先 `SELECT 1` 探活，发现死连接立即丢弃并新建——**这是本地开发跨公网连云 MySQL 时避免 500 的关键**。
  - `pool_recycle=1800` 30min 回收，是次级防御。
  - **真实根因（2026-07-19 踩坑）**：本地开发机 → 云 MySQL（`1.12.219.195:3306`）跨公网，中间的 NAT/防火墙会**在连接空闲几分钟后静默掐断 TCP**（MySQL 自身 `wait_timeout=28800s` 远大于此，所以不是 MySQL 超时）。进程启动时建的连接空闲一会儿后即成「死连接」，下一次查询直接抛 `(2013) Lost connection to MySQL server during query` → 注册/登录/建会话/落库对话等一切写库接口 500。此前 `db.py` 漏写 `pool_pre_ping`（文档早就要求，属代码/文档漂移），于 v0.4.5 补上后 500 消失。Redis 与此无关（用量统计写 Redis 正常）。
  - 生命周期：`lifespan` 启动建引擎，关闭时 `await engine.dispose()`。
- **Redis（redis.asyncio 连接池）**：
  - `pool = redis.asyncio.ConnectionPool.from_url(REDIS_URL, max_connections=50, decode_responses=True, health_check_interval=30)`；客户端 `redis.asyncio.Redis(connection_pool=pool)` 全局复用。
- **统一约束**：客户端均在服务启动时初始化为模块级单例，关闭时释放；**禁止在请求处理函数内反复 `create_engine` / 新建连接**。
