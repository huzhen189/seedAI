# v0.8.2 方案：系统 Prompt 与项目系统 Prompt 的接通与职责澄清

> 状态：**已实施（v0.8.2，2026-07-21 落地）**
> 关联：`v0.8.1-意图管道排查与优化方案.md`（意图管道决策链路）
> 目标：澄清 `run_rules` / `INTENT_SYSTEM` / `project.system_prompt` 的职责边界，并修复「项目系统 prompt 是死数据」的缺口。

---

## 0. 背景与问题

用户提出两个疑问：

1. `run_rules` 好像没有按系统 prompt / 项目系统 prompt 过滤，是不需要吗？
2. 系统 prompt 和项目系统 prompt 用在哪个流程？

经代码核查，结论如下。

### 0.1 当前三层 prompt 的真实位置

| 名称 | 定义位置 | 消费位置 | 现状 |
|------|----------|----------|------|
| **全局系统 prompt** `INTENT_SYSTEM` | `backend/ai_service/app/intent/semantic.py:21` | **仅** `run_semantic()`（`intent/semantic.py:89` 作为 `role:system` 发给分类 LLM） | ✅ 生效，用于**意图分类流程** |
| **规则模块** `run_rules` | `backend/ai_service/app/intent/rules.py:25` | `pipeline.py:66`，输出 `RuleResult` | ✅ 生效，但**零 LLM 关键词快检**，故意不读任何 prompt |
| **安全模块** `run_safety` | `backend/ai_service/app/intent/safety.py:40` | `pipeline.py:68`，输出 `SafetyResult`（`risk_level` 驱动 block/confirm） | ✅ 生效，但**硬编码关键词集**，不读 prompt |
| **项目系统 prompt** `project.system_prompt` | `backend/business/app/models.py:51`（字段）+ `projects.py:390/403`（`get/update_project_prompt` API） | **无** | ❌ **死数据**：business 可存可取，ai_service 零引用 |

### 0.2 关键结论

- **`run_rules` 不读 prompt 是正确设计，不是 bug。**
  `run_rules` 是零 LLM 的词法快捷分类（build/code/learn + 行业），与 `INTENT_SYSTEM`（喂 `run_semantic` 的 LLM 分类器 prompt）是**互补信号**，二者在 `pipeline._aggregate` 里按权重融合（语义 70% + 规则 20% + 上下文 10%）。让 `run_rules` 去"读 prompt"既冗余又会造成循环依赖。

- **真正的缺口是 `project.system_prompt` 整条链路断了。**
  `queue.py` 的 job 只带 `project_status`，**没有 `project_id` / `system_prompt`**；ai_service 没有任何文件引用 `project_system_prompt`；所有 skill 的 `SYS_*` prompt 都是硬编码（含行业特化，但不含项目定制）。
  → 用户在项目里配置的「项目系统 prompt」**从来没生效过**。

- **潜在一致性债务**：`INTENT_SYSTEM` 里的「unsupported 类目」（非网页前端需求 → unsupported）与 `run_safety` 的 `CRITICAL/HIGH` 关键词集是两份手写来源，存在漂移风险（分类器说 unsupported 但安全层放行，或反之）。

## 2. Tier 0 · 职责边界澄清（本文档即交付）

明确约定，写进开发文档，后续评审以本文为准：

- **`INTENT_SYSTEM`（全局系统 prompt）**：意图分类器的指令集，**只喂 `run_semantic`**。定义 taxonomy（level1/level2/industry）、unsupported 规则、断点关系。
- **`run_rules` / `run_safety`（零 LLM 硬规则）**：即时结构性检查，**故意不读 prompt**。`run_rules` 出粗分类，`run_safety` 出风险等级（critical/high/medium/low）。
- **`project.system_prompt`（项目系统 prompt）**：项目级自定义指令，**归属「skill 执行上下文」**，在 skill 构造 `SYS_*` 时注入，不归规则层。
- 三者通过 `pipeline._aggregate` 融合决策，互不从属。

## 3. Tier 1 · 接通项目系统 Prompt（P0，修复死功能）

### 3.1 目标
让项目的 `system_prompt` 随对话下发到 ai_service，并注入 skill 的 system prompt，使生成结果遵循项目定制。

### 3.2 数据流
```
前端(带 project_id)
  → business /api/chat (proxy.py)
      ├─ 已有: project_status
      └─ 新增: get_project_prompt(project_id) → system_prompt
  → job payload 新增字段: project_system_prompt
  → ai_service /generate (queue.py)
      └─ job["project_system_prompt"] 透传
  → run_skill(..., project_system_prompt)
      └─ 各 skill 构造 SYS_* 时追加「项目约束」段
```

### 3.3 改动点（伪代码，实施时细化）

**① business `proxy.py`（生成转发前）**
```python
# 已有 project_status；补充 project_id 与 system_prompt
project_id = payload.get("project_id")
project_system_prompt = ""
if project_id:
    pr = await get_project_prompt(project_id)      # projects.py:390
    project_system_prompt = pr.get("system_prompt", "") or ""
payload["project_system_prompt"] = project_system_prompt
# 注意: 前端是否已把 project_id 传给 /api/chat 需先确认(当前 job 有 project_status，可能未传 project_id)
```

**② ai_service `queue.py`**
```python
proj_prompt = job.get("project_system_prompt", "")   # 新增
# 透传到 run_skill(..., project_system_prompt=proj_prompt)
```

**③ ai_service skills（以 `generate_site.py` / `requirement_agent.py` 为例）**
```python
def _build_sys(skill_base_sys: str, project_system_prompt: str) -> str:
    if not project_system_prompt:
        return skill_base_sys
    return skill_base_sys + "\n\n# 项目约束(用户定制，必须遵循)\n" + project_system_prompt
# generate_site: SYS_CODER = _build_sys(SYS_CODER, project_system_prompt)
# requirement_agent: full_sys = _build_sys(full_sys, project_system_prompt)
```

### 3.4 前置确认项
- 前端 → business → ai_service 的链路里，`project_id` 是否已在某处传递？当前 `job` 只见 `project_status`，需确认 `project_id` 来源（会话绑定项目？前端 chat 请求带？），补齐传递。
- `get_project_prompt` 需 `async`，proxy 当前是否已在 async 上下文（是，`proxy.py` 为 async 生成器）。

### 3.5 验证
1. 起项目，在「项目设置」填 `system_prompt`（如"所有页面主色用深蓝，文案口吻正式"）。
2. 对话触发 `generate_site` / `requirement_agent`。
3. 确认生成产物遵循该约束（颜色/口吻符合）。
4. 无 `system_prompt` 的项目行为不变（向后兼容）。

## 4. Tier 2 · 项目级约束进安全层（P1，可选）

### 4.1 适用场景
当项目需要**硬性禁止某些意图/话题**（如"禁止部署类操作""屏蔽某行业"），不能只依赖 LLM 听话，应在 `run_safety` 层面拦截/确认。

### 4.2 关键约束（防坑）
**不要把整段自然语言 `system_prompt` 直接塞进规则引擎**——自由文本不可控，易误拦/漏拦。
改为**结构化字段**：新增 `project_constraints`（或约定 `system_prompt` 内含机器可读行，如 `--forbid: deploy, payment`）。

### 4.3 改动点（伪代码）
```python
# intent/safety.py
def run_safety(messages, project_constraints: list[str] | None = None) -> SafetyResult:
    ...
    if project_constraints:
        for kw in project_constraints:
            if kw in t:
                return SafetyResult(risk_level="critical", ..., block_reason=f"项目禁止: {kw}")
```
- `classify_v2` / `pipeline` 需把 `project_constraints` 传进 `run_safety`。
- `project_constraints` 由 business 从项目配置抽取后随 job 下发（同 Tier 1 的 `project_system_prompt` 通道）。

## 5. Tier 3 · `INTENT_SYSTEM` ↔ `run_safety` 一致性（P2）

### 5.1 问题
`INTENT_SYSTEM` 的 unsupported 描述与 `run_safety` 关键词集是两份手写来源，修改其一易漏另一处。

### 5.2 方案
在 `intent/common.py` 增加单一来源常量：
```python
# 全局"非网页前端/禁用"类目（分类器与规则引擎共享）
UNSUPPORTED_HINTS = {...}      # 用于 INTENT_SYSTEM 文本生成
FORBIDDEN_KEYWORDS = {...}     # run_safety 引用
```
- `INTENT_SYSTEM` 文本由 `UNSUPPORTED_HINTS` 渲染生成（避免硬编码字符串）。
- `run_safety.CRITICAL/HIGH` 引用 `FORBIDDEN_KEYWORDS`。
- 新增禁用类目只需改 `common.py` 一处。

## 6. 风险与注意

| 项 | 风险 | 规避 |
|----|------|------|
| Tier 1 透传 prompt | prompt 过长撑爆上下文 | 业务侧已限制 `system_prompt` ≤ 4000 字符（`projects.py:416`），skill 注入时注意总长度 |
| Tier 2 自由文本进规则 | 误拦/漏拦 | 仅用结构化 `project_constraints`，不解析自然语言 |
| project_id 未传 | Tier 1 无法落地 | 先确认 `project_id` 链路（见 3.4） |
| 向后兼容 | 老项目无 system_prompt | 空值分支保持原行为，不报错 |

---

## 7. 建议实施顺序

1. **先确认 `project_id` 是否在 chat 链路中可用**（决定 Tier 1 入口）。
2. 实施 **Tier 1**（P0，修复死功能，用户感知最强）。
3. 视需求实施 **Tier 2**（P1，项目硬约束）。
4. 择机做 **Tier 3**（P2，一致性，低紧急）。
