# 记忆压缩方案（短期记忆 → 长期记忆）

> 目的：把"只 append 的流水"变成"精炼可检索的事实库"，让每次新会话的上下文预算花在刀刃上。
> 适用范围：workspace 记忆（`.workbuddy/memory/`）+ user 级记忆（`~/.workbuddy/MEMORY.md`）。Cloud 记忆（Layer 1）由服务端托管，不在本方案内。

|---|---|---|---|---|

**结论**：从未压缩，只 append。根因是习惯错了——把"该进长期记忆的结论"重复堆在每日日志，又把长期记忆当"流水备份"而非"精炼事实库"。

## 1. 分层定义

- **短期记忆** = 每日日志 `YYYY-MM-DD.md`：原始、append-only、易膨胀，是压缩的源。
- **长期记忆** = 两个 `MEMORY.md`：
  - workspace（上限 **3000** 字符）：本项目长期约定/架构/功能状态。
  - user 级（上限 **4000** 字符）：跨项目个人习惯/硬约束。

## 2. 压缩触发条件（Trigger）

1. **硬删（现有规则）**：每日日志 mtime 距今 **≥ 30 天** → 蒸馏进 `MEMORY.md` 对应 topic → **删除旧日志文件**。
2. **软蒸馏**：单日志 **> 120 行 或 > 6000 字符** → 当次会话收尾前，把增量结论写进 `MEMORY.md`（**不删**日志，保留溯源）。
3. **预算守卫**：任一 `MEMORY.md` 超自身上限 → 触发 prune 巡检（见 §4）。
4. **收尾钩子（纠正旧习惯）**：每次收尾，把"本次有价值的结论"直接写进 `MEMORY.md` 对应 topic，而非只写每日日志。

## 3. 蒸馏规则：价值分层

- **T0 永久保留**：用户硬约束（文件复制铁律、不自动 push、记忆分层、身份覆盖指令）、架构/端口、当前大版本状态线。
- **T1 长期保留**：功能落地状态、已知技术债 backlog、长期约定（如"改动即同步 docs/01~04"）。
- **T2 易逝丢弃**：已修复且已 commit 的具体 bug 故事、具体 commit SHA（里程碑除外）、临时路径、agent 内部报错、"进行中"标记（完成即删）。
- **压缩手法**：长叙事 → 一句话事实。例：10 行 bug 经过 → `v0.4.4 误判 Redis，真因 MySQL 跨公网断连，v0.4.5 用 pool_pre_ping+pool_recycle 修`。

## 4. 预算执行（硬卡）

- `MEMORY.md` 超限时，按 **T2 → T1 细节** 顺序删，保 T0。
- 用 `wc -c` 卡 3000（workspace）/ 4000（user）红线，超标即拦、不得继续 append。

## 5. 自动化（set-and-forget）

- 把压缩逻辑固化成 **skill `memory-compactor`**（可手动触发、可被 automation 调用）。
- 建 **daily automation**（每天 03:00）：
  1. 扫描 ≥30 天每日日志 → 蒸馏进 `MEMORY.md` → 删除旧日志；
  2. 检查两个 `MEMORY.md` 字符数 → 超预算则 prune；
  3. 把"压缩报告"写回当日日志一行（不破坏 append-only 语义）。
