# AI 生成器平台 — 开发文档 & 架构方案 (v0.1)

> 版本:v0.1(草稿,持续共创)
> 最后更新:2026-07-17
> 一句话定位:一个通过 AI 对话即可生成**网站/前端页面**(后续扩展文档、代码)的个人平台,可小范围分享给朋友使用。

---

## 1. 项目概述

### 1.1 产品定位
- **形态**:类 v0.dev / bolt.new / Lovable 的"对话即生成"平台。
- **使用范围**:自己为主 + 小范围分享(朋友/小圈子),需要**轻量账号体系**和**用量限制**,但不做完整商业化。
- **首要产物**:网站 / 前端页面(带实时预览、可分享、可迭代)。
- **后续扩展**:文档生成、代码工程生成(架构预留,不在 MVP)。

### 1.2 核心价值
用户用自然语言描述需求 → AI 生成可运行的网页 → 实时预览 → 继续对话迭代 → 一键分享/导出。

---

## 2. 核心功能范围

### 2.1 MVP(第一阶段必须有)
| 模块 | 说明 |
|------|------|
| 对话生成 | 输入需求,流式返回生成过程与结果 |
| 实时预览 | 生成的页面在 iframe / 沙箱中即时渲染 |
| 迭代修改 | 基于已有结果继续对话("把按钮改成蓝色") |
| 项目管理 | 保存每次生成为"项目",可重新打开 |
| 轻量账号 | 登录 + 每人用量限制(防止 Key 被刷爆) |
| 分享链接 | 生成只读预览链接分享给朋友 |

### 2.2 第二阶段(增强)
- 版本历史与回滚
- 代码在线编辑(改完实时预览)
- 导出为 zip / 一键部署
- 模板库(常见页面起手式)

### 2.3 暂不做(明确排除)
- 完整多租户 / 订阅计费
- 复杂后端逻辑生成(数据库、API)
- 团队协作 / 权限体系

## 3. 技术选型

| 层 | 选型 | 理由 |
|----|------|------|
| 全栈框架 | **Next.js (App Router) + TypeScript** | 一套 TS 前后端,部署简单,AI 流式生态成熟 |
| UI | Tailwind CSS + shadcn/ui | 快速搭建、风格统一 |
| AI 调用 | **Vercel AI SDK** + 云端大模型 API | 统一封装流式输出,方便切换模型 |
| 模型 | 云端 API(见 §3.1 待定) | 效果好、上手快、按量付费 |
| 预览沙箱 | iframe srcdoc(MVP) → Sandpack(增强) | 见 §5 |
| 数据库 | ⬜ 待定(见 §3.2) | 存项目/对话/用户 |
| 鉴权 | ⬜ 待定(见 §3.2) | 轻量登录 |
| 部署 | Vercel(推荐) 或 自建 Node | 小范围分享,Vercel 免运维 |

### 3.1 ⬜ 待决策:主力模型
你已选"调用云端大模型 API",具体用哪家还没定。候选:
- **DeepSeek**:代码能力强、国内访问稳、便宜。
- **Claude (Anthropic)**:生成网页/前端质量目前第一梯队,但需海外网络。
- **通义千问 / 豆包 / Kimi**:国内合规、稳定。
- **方案建议**:用 AI SDK 做**模型抽象层**,默认接 1 家,保留随时切换的能力。

### 3.2 ⬜ 待决策:数据库 + 鉴权
- **方案 A — Supabase**:自带 Postgres + Auth + 存储,省事,免费额度够个人用。(推荐,小范围分享最快)
- **方案 B — Vercel Postgres + Auth.js**:全在 Vercel 生态,和 Next.js 贴合。
- **方案 C — 本地 SQLite + 简单 Token**:纯自用最简,但分享给朋友时略麻烦。

## 4. 系统架构

```
┌─────────────┐   对话/需求    ┌──────────────────┐
│   浏览器端    │ ────────────▶ │  Next.js 服务端    │
│ (对话 UI +   │ ◀──────────── │  (API Routes)     │
│  预览面板)    │   流式结果     └────────┬─────────┘
└──────┬──────┘                          │
       │                          ┌──────▼──────┐
       │ iframe/Sandpack          │ 大模型 API   │
       │ 渲染生成结果               │ (AI SDK)    │
       ▼                          └──────┬──────┘
┌─────────────┐                          │
│  预览沙箱     │                   ┌──────▼──────┐
└─────────────┘                   │  数据库/鉴权  │
                                  │ 项目/对话/用户 │
                                  └─────────────┘
```

**关键约束**:模型 API Key 只放在**服务端**,浏览器永远不接触 Key,所有 AI 调用经 Next.js API Route 中转。

## 5. 核心技术难点:生成结果的预览与运行

这是这类产品真正的门槛,分场景处理:

| 生成内容类型 | 预览方案 | 复杂度 |
|-------------|---------|--------|
| 纯 HTML/CSS/JS 单页 | `<iframe srcdoc="...">` 直接注入 | ⭐ 低(MVP 首选) |
| 含 React/多文件/需构建 | **Sandpack**(CodeSandbox 浏览器内打包) | ⭐⭐ 中 |
| 需真实 Node 运行环境 | WebContainer(StackBlitz) 或服务端沙箱 | ⭐⭐⭐ 高(暂不做) |

**MVP 决策**:先只生成**单文件 HTML(内联 CSS/JS,可用 CDN 引 Tailwind/React UMD)**,用 iframe srcdoc 预览。这样 0 构建、0 沙箱成本,当天就能跑通闭环。增强阶段再上 Sandpack 支持多文件工程。

## 7. 开发路线图

- **M0 打通闭环(核心)**:对话 → 调模型 → 生成单文件 HTML → iframe 预览。不含登录、不含持久化。
- **M1 可用**:加数据库 + 轻量登录 + 项目保存/列表 + 用量限制。
- **M2 可分享**:分享只读链接 + 迭代修改(基于上下文改)。
- **M3 增强**:Sandpack 多文件、版本历史、导出 zip、模板库。
