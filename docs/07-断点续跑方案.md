## 三、断连 => 保存断点

### 3.1 检测断连

`proxy.py` publisher 生成器在每帧 yield 后调用 `await request.is_disconnected()`：

```python
async def publisher():
    for frame in generate_frames():
        yield frame
        if await request.is_disconnected():
            await notify_cancel(trace_id)   # 设 Redis cancel 标记
            break
```

AI Worker 的 `is_cancelled()` 回调读 Redis cancel 标记。**零轮询开销**，复用现有取消机制。

### 3.2 `generate_site.py` 保存 checkpoint

每个阶段之间检查 `is_cancelled()`——检测到取消时：

```python
async def _save_checkpoint(trace_id, stage, data, conversation_id, db):
    conv = await db.get(Conversation, conversation_id)
    conv.status = "paused"
    conv.checkpoint_stage = stage
    conv.checkpoint_data = json.dumps(data, ensure_ascii=False)
    conv.progress_pct = STAGE_PCT[stage]
    await db.commit()
```

| stage | progress_pct | checkpoint_data 内容 |
|-------|-------------|---------------------|
| `planner_done` | 25 | `{plan, rag_ctx, messages}` |
| `coder_done` | 65 | `{plan, html, rag_ctx, messages}` |
| `reviewer_r1` | 75 | `{plan, html, attempt:0}` |
| `reviewer_r2` | 85 | `{plan, html, attempt:1}` |
| `reviewer_r3` | 95 | `{plan, html, attempt:2}` |

### 3.3 paused 事件

保存成功后 yield 一个 `paused` SSE 事件：

```python
yield ev("paused", stage=stage, progress=progress_pct, checkpoint_stage=stage)
yield ev("done")
```

前端收到 `paused` → 设置 `onPaused` 回调 → 正常结束 SSE（不显示错误）。

## 五、新对话 vs 断点：LLM 关系判定

### 5.1 触发时机

`ChatView.send()` — 用户敲回车时，如果当前项目存在 `status == "paused"` 的会话。

### 5.2 五级关系

| 关系 | 含义 | 判定方式 |
|------|------|---------|
| `unrelated` | 在其他会话里打字 | `convId != pausedConv.id`，前端直接判 |
| `resume` | 从断点继续 | LLM（搭载 intent_classifier） |
| `correct` | 在旧基础上改 | LLM |
| `override` | 放弃旧结果重来 | LLM |
| `unclear` | LLM 无法判断 | LLM，前端弹框四选一 |

### 5.3 判定方式：搭载 intent_classifier

每次对话必调 `classify_intent()`，在同一个 LLM 调用的 prompt 末尾追加约 80 token：

```
如果当前项目存在一个未完成的生成(断点为 {checkpoint_stage}, 已完成 {progress_pct}%),
判断用户新消息和断点的关系, 返回 checkpoint_relation(5选1):

- resume: 要继续上次的工作(如 "继续""接着做""go on")
- correct: 在旧基础上调整(如 "改导航""换颜色""优化一下")
- override: 放弃旧结果重来(如 "重新做""不要之前的内容")
- unrelated: 说另一件事, 和断点无关
- unclear: 无法判断

返回 JSON 增加 checkpoint_relation 字段
```

一次 LLM 调用同时出四个维度：

```json
{
  "level1": "build",
  "level2": "modify",
  "confidence": 0.92,
  "industry": "restaurant",
  "checkpoint_relation": "correct"
}
```

**零额外延迟**——断点关系判定搭载在同一次分类调用上。

### 5.4 各关系处理流程

```
unrelated → 正常 send(), 断点保持 paused
resume    → 加载 checkpoint_data → 从对应阶段 resume
override  → 清 status="active" + checkpoint=NULL → 正常 send()
correct   → plan.goal += "更正: {新输入}" → checkpoint.stage 回退一级 → resume
unclear   → 弹框: [继续上次] [在此基础上修改] [重新开始] [新对话]
```

### 5.5 流程图

```
用户发消息
  │
  ├─ 本项目无 paused ──────────→ 正常 send()
  │
  ├─ convId != pausedConv.id ──→ unrelated → 正常 send()
  │
  └─ convId == pausedConv.id
       └─ LLM 分类出 checkpoint_relation
            ├─ resume   → 从断点继续
            ├─ correct  → 注入更正, resume(mode=correct)
            ├─ override → 清 checkpoint, 正常 send()
            ├─ unrelated→ 正常 send()
            └─ unclear  → 弹框四选一
```

## 六、Agent 完整恢复方案

### 6.1 `generate_site.py` resume 入口

```python
async def generate_stream(model_id, messages, trace_id=None,
                          checkpoint=None, resume_mode="resume", ...):
    if checkpoint:
        stage = checkpoint["stage"]
        plan = checkpoint["plan"]
        html = checkpoint.get("html", "")
        attempt = checkpoint.get("attempt", 0)

        if resume_mode == "correct":
            # 更正模式: 回退一级 (已有的产出保留, 让 AI 在此基础上改)
            if stage.startswith("reviewer_r"):
                stage = "coder_done"
                attempt = 0
            # coder_done → 保持, 带旧 html 重新执行 Coder
            # planner_done → 不处理 (前端已判为 override)

        if stage == "coder_done":
            yield ev("node", stage="enter_coder_done")  # 标记已跳过
            yield from _reviewer_loop(model_id, html, plan, 0, is_cancelled)

        elif stage in ("reviewer_r1", "reviewer_r2", "reviewer_r3"):
            attempt = int(stage[-1])  # "reviewer_r2" → 2
            yield ev("node", stage=f"resume_{stage}")
            yield from _reviewer_loop(model_id, html, plan, attempt, is_cancelled)

        elif stage == "planner_done":
            yield ev("node", stage="enter_planner_done")
            yield from _coder(model_id, plan, messages, is_cancelled)
            yield from _reviewer_loop(model_id, html, plan, 0, is_cancelled)

    else:
        # 正常流程: Planner → Coder → Reviewer → 预览 → done
        ...
```

### 6.2 各阶段恢复起点

```
checkpoint_stage    →  恢复后执行的第一个动作
──────────────────────────────────────────
planner_done        →  Coder（plan 已就绪, 重新生成 HTML）
coder_done          →  Reviewer round 1（html 已就绪）
reviewer_r1         →  Reviewer round 2
reviewer_r2         →  Reviewer round 3
reviewer_r3         →  预览投递 + 完成
```

### 6.3 correct 模式特殊处理

| 断点阶段 | correct 后从哪开始 |
|---------|------------------|
| `planner_done` | 同 override（从头来） |
| `coder_done` | Coder 重新执行，注入旧 html + 更正指令 |
| `reviewer_rN` | 回退到 Coder，注入旧 html + 更正指令 |

### 6.4 恢复后的收尾

恢复流程完成后，`proxy.py` 的 `_persist_conversation` 正常执行：
- `checkpoint_data` → NULL
- `status` → `completed`
- `progress_pct` → `100`
- user + assistant 消息正常写入 `messages`
