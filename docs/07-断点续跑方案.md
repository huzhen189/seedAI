### 3.2 保存 checkpoint → conversations 表

`generate_site.py` 在每个阶段之间调用 `is_cancelled()`, 检测到取消时:

```python
async def _save_checkpoint(trace_id, stage, data, db_session):
    conv = (await db.execute(
        select(Conversation).where(Conversation.id == conv_id)
    )).scalar_one_or_none()
    if conv:
        conv.status = "paused"
        conv.checkpoint_stage = stage
        conv.checkpoint_data = json.dumps(data, ensure_ascii=False)
        conv.progress_pct = STAGE_PCT[stage]
        await db_session.commit()
```

| stage | progress_pct | data 包含 |
|-------|-------------|----------|
| `planner_done` | 25 | `{plan, rag_ctx, messages}` |
| `coder_done` | 65 | `{plan, html, rag_ctx, messages}` |
| `reviewer_r1` | 75 | `{plan, html, attempt:0}` |
| `reviewer_r2` | 85 | `{plan, html, attempt:1}` |
| `reviewer_r3` | 95 | `{plan, html, attempt:2}` |

### 3.3 paused 事件

保存成功后 yield:

```python
yield ev("paused", stage=stage, progress=progress_pct, checkpoint_stage=stage)
yield ev("done")
```

前端收到 `paused` → 设置 `onPaused` 回调 → 不显示错误, 等下次进来恢复。

## §5 新对话与断点的关系判定

### 5.1 判定时机

`ChatView.send()` — 用户敲回车发送消息时, 如果当前项目存在 `status == "paused"` 的会话, 执行关系判定。

### 5.2 五级关系分类

| 关系 | 含义 | 典型输入 | 触发条件 |
|------|------|---------|---------|
| `resume`（延续） | 从断点继续做 | 用户点"继续生成"按钮 | 显式触发 |
| `override`（覆盖） | 完全重来，不要旧东西 | "重新做一个""不要之前的" | 关键词命中 |
| `correct`（更正） | 基于断点调整，保留已产出 | "导航栏改了""把底色换成白色" | 关键词命中 |
| `unrelated`（无关） | 和断点没任何关系 | 在其他会话里打字 | `convId != pausedConv.id` |
| `ask`（问用户） | 不确定关系，弹框让选 | 无法自动判断时 | 默认兜底 |

### 5.3 自动判定逻辑（前端纯关键词）

```typescript
function classifyPausedRelation(newText: string, pausedConv: Conversation): string {
  // 1) 不同会话 → 无关
  if (currentConvId !== pausedConv.id) return "unrelated"

  // 2) 同一会话: 关键词判定
  const t = newText.toLowerCase()
  
  // override: 明确要重来
  if (/重新|重来|不要|取消|算了|换一个|从头/.test(t)) return "override"
  
  // correct: 修改/调整型 (checkpoint 阶段 > planner_done 才有意义)
  if (/改|换|调整|修正|不对|应该是|换成|修改|优化/.test(t)) {
    // 有已产出的 html 才更正, planner_done 阶段更正=覆盖
    return pausedConv.checkpoint_stage !== "planner_done" ? "correct" : "override"
  }

  // 3) 默认: 弹框问用户
  return "ask"
}
```

### 5.4 各关系的处理流程

```
┌─ resume（延续）
│   加载 checkpoint_data → 从对应阶段 resume → 同上流程
│
├─ override（覆盖）
│   清 conv.status="active" + checkpoint_data=NULL
│   → 正常新建 send() 流程（新的 trace_id）
│
├─ correct（更正）
│   保留 checkpoint_data
│   → 注入更正指令到 messages: {"role":"user","content":"额外要求: 请在上次基础上调整，{新输入}"}
│   → resume 流程, 但 checkpoint_data.plan.goal 追加更正说明
│   → 如果是 coder_done 阶段: Coder 重新执行（带旧 html + 更正指令）
│   → 如果是 reviewer_rN 阶段: 退回到 Coder 重新执行
│
├─ unrelated（无关）
│   不做任何操作, 正常 send()
│
└─ ask（问用户）
    弹 confirm 对话框:
      "检测到未完成的生成「{pausedConv.title}」(已完成 {progress_pct}%)"
      [继续上次] → resume
      [在此基础上修改] → correct
      [重新开始] → override
      [这是新对话] → unrelated（自动切换到新会话）
```

### 5.5 AI 侧对 correct 的处理

`proxy.py` 收到 `resume=true&correct=true` 时:

```
从 conversations 读 checkpoint_data
  → plan.goal += "\n更正指令: {user新输入}"
  → 注入新 user message: "请在上次产出的 HTML 基础上调整: {user新输入}"
  → checkpoint_data.stage 回退一级:
      reviewer_rN → coder_done  (重新 Coder)
      coder_done → coder_done   (Coder 带旧 html + 更正指令)
      planner_done → 等同于 override (从头来)
  → 传给 generate_site.py resume 入口
```

### 5.6 流程图

```
用户发消息
  │
  ├─ 本项目无 paused 会话 ──→ 正常 send()
  │
  ├─ convId != pausedConv.id ──→ unrelated ──→ 正常 send()
  │
  └─ convId == pausedConv.id
       │
       ├─ 关键词"重新/重来/不要" ──→ override ──→ 清 checkpoint, 正常 send()
       │
       ├─ 关键词"改/调整/修正" ──→ correct ──→ 注入更正指令, resume(mode=correct)
       │
       └─ 无法判定 ──→ ask ──→ 弹框四选一 ──→ 用户选后执行
```

## §6 Agent 完整恢复方案

### 6.1 `generate_site.py` resume 入口

```python
async def generate_stream(model_id, messages, trace_id=None,
                          is_cancelled=None, checkpoint=None,
                          resume_mode="resume",  # "resume" | "correct"
                          intent=None, ...):
    if checkpoint:
        stage = checkpoint.get("stage")
        plan = checkpoint.get("plan", {})
        html = checkpoint.get("html", "")
        attempt = checkpoint.get("attempt", 0)

        # correct 模式: 回退一级
        if resume_mode == "correct":
            if stage == "reviewer_r1":  stage = "coder_done"; attempt = 0
            elif stage == "reviewer_r2": stage = "coder_done"; attempt = 0
            elif stage == "reviewer_r3": stage = "coder_done"; attempt = 0
            # coder_done → 保持, planner_done → 不处理(前端已判为 override)

        if stage in ("coder_done", "reviewer_r1", "reviewer_r2", "reviewer_r3"):
            yield ev("node", stage="enter_coder_done")
            if stage == "coder_done":
                attempt = 0
            else:
                attempt = int(stage[-1])  # "reviewer_r2" → 2
            yield from _reviewer_loop(model_id, html, plan, attempt, is_cancelled)

        elif stage == "planner_done":
            yield ev("node", stage="enter_planner_done")
            yield from _coder(model_id, plan, messages, is_cancelled)
            yield from _reviewer_loop(model_id, html, plan, 0, is_cancelled)

    else:
        # 正常流程
        ...
```

### 6.2 各阶段恢复起点

```
checkpoint_stage       →  恢复起始动作
─────────────────────────────────────────────
planner_done           →  重新执行 Coder（plan 已就绪）
coder_done             →  跳过 Coder → 直接进 Reviewer round 1
reviewer_r1            →  跳过 Coder → 进 Reviewer round 2
reviewer_r2            →  跳过 Coder → 进 Reviewer round 3
reviewer_r3            →  直接进 预览投递 + 完成
```

### 6.3 恢复后的完整路径

```
planner_done 恢复:
  → Coder(用 plan) → Reviewer(r1→r2→r3) → 预览 → COS → done

coder_done 恢复:
  → Reviewer(r1→r2→r3) → 预览 → COS → done

reviewer_rN 恢复:
  → Reviewer(rN+1→r3) → 预览 → COS → done
```

### 6.4 恢复后的消息落库

恢复流程完成后，`proxy.py` 的 `_persist_conversation` 正常执行：
- `checkpoint_data` 清空
- `status` → `completed`
- `progress_pct` → `100`
- user/assistant 消息正常写入 `messages`
